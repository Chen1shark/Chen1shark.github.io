操作系统

进程

进程运行态转就绪态

进程状态转换中，运行态→就绪态的常见触发条件：

  • 时间片用完：分时系统中，进程的 CPU 时间片耗尽，被调度器强制切换回就绪队列。
  • 更高优先级进程就绪：在抢占式调度中，若有更高优先级的进程进入就绪态，当前运行进程被抢占，退回就绪态。

运行态→就绪态是被动切换（非自愿让出 CPU），与运行态→阻塞态（主动等待 IO 等资源）不同。

虚拟内存和物理内存

页、页框、页表

概念 说明
页（Page） 虚拟地址空间的最小划分单位（通常 4KB），是逻辑上的概念
页框（Frame） 物理内存的最小划分单位，大小与页相同
页表（Page Table） 记录虚拟页→物理页框映射关系的数据结构，每个进程一张
页表项（PTE） 页表中的一条记录，包含物理页框号、有效位、权限位等

页是页表的管理对象

这里的页特指虚拟页，它是进程虚拟地址空间的最小划分单位。页表不是为某一个虚拟页单独存在，而是为整个进程的虚拟地址空间服务，其核心功能就是记录进程中需要映射的虚拟页，分别对应哪一个物理页框。只有存在映射需求的虚拟页，才会在页表中拥有对应的页表项；未使用或被换出的虚拟页，在页表中无有效页表项或标记为无效状态。

页表是虚拟页与物理页框的映射桥梁

虚拟页本身不对应实际物理存储，必须通过页表的页表项，才能建立与物理页框的关联。页表项中存储的物理页框号，是虚拟页能够被 CPU 访问的关键；同时页表项中的权限位、有效位等字段，还会直接管控虚拟页的访问规则（如可读可写、只读）和映射状态（如已映射、换出到磁盘）。

内存空间分配

连续分配场景题

已知有 100MB 空间，A 操作使用了 30MB，然后 B 操作使用了 35MB，然后删除了 A 操作的占用，然后 C 操作使用了 45MB，然后删除了 B 操作的占用，求最大空闲空间。

关键步骤：C 操作时，连续内存不够用，被分成了 B 操作前面的 30MB 和 B 操作后的 15MB，因此最后总空间为 55MB，最大空闲连续空间为 35MB。

Windows 相关

本地安全与域安全策略

核心定位与作用范围关系

Windows 本地安全策略与域安全策略是层级互补的安全管控体系，本地安全策略作用于单台计算机，管控本地账户与单机安全规则，适配工作组等分散环境；域安全策略以域控制器为核心，可集中管控域内所有成员设备的安全配置，适配中大型企业的统一管理需求。二者仅在计算机加入域时产生交互，未加入域的设备仅受本地安全策略约束。

优先级与执行规则关系

二者的核心执行原则是域安全策略优先于本地安全策略：计算机加入域后，若两套策略存在冲突，域策略会覆盖本地策略并锁定相关配置项；若策略无冲突，则会合并生效。同时，域策略不会完全替代本地策略，未被域策略定义的本地安全规则仍可生效，可作为域策略的补充满足单台设备的个性化安全需求；而域控制器的本地策略会升级为域控制器安全策略，优先级高于普通域安全策略。

域策略的优先级更高，是为了实现"全局统一管控"，而不是因为范围大小；同层级的策略里，才遵循"小范围优先于大范围"的逻辑。

Java 基础

类变量和静态变量

实际为同一概念：

  • 静态变量：通过 static 关键字修饰的变量（语法层面的定义）。
  • 类变量：属于"类本身"而非类的实例对象的变量（语义层面的定义）。

类变量作用域

  • 误区 1：“类变量作用域和实例绑定”。❌ 类变量不归属于实例，哪怕没有创建任何对象，只要类加载完成，就可通过 类名.变量名 访问（比如 Math.PI，无需 new Math()）。✅ 作用域是"类本身"，实例仅能间接访问，而非决定作用域。
  • 误区 2：“静态方法能访问所有类的变量”。❌ 静态方法可访问「当前类的类变量」，但访问「其他类的类变量」仍受访问修饰符约束（比如其他类的 private 静态变量无法访问）。✅ 静态方法的访问权限 ≠ 类变量的作用域，作用域最终由变量自身的修饰符决定。
  • 误区 3：“类变量作用域随方法执行结束消失”。❌ 类变量存储在方法区，生命周期和类绑定，而非方法；哪怕所有方法执行完毕，类变量仍存在（除非类被卸载）。

异常处理

异常抛出的作用域

由方法 / 代码块的可见性 + 声明约束：

  • 检查型异常（如 IOException）：抛出范围受 throws 声明约束，仅能在声明了该异常的方法 / 子类重写方法中抛出（子类重写时不能抛出父类未声明的检查型异常，符合里氏代换原则）。
  • 运行时异常（如 NullPointerException）：无显式声明约束，可在任意代码块（方法、循环、分支）中抛出，作用域覆盖当前执行的代码路径。
  • 异常的传播作用域：未捕获的异常会沿调用栈向上传播（从当前方法→调用方方法→上层调用方），直到被捕获或终止线程，传播范围不跨线程（线程内独立传播）。

异常捕获的作用域

仅覆盖「try 代码块内的执行逻辑」。catch 块仅能捕获对应 try 块内执行过程中抛出的、匹配异常类型（或子类类型）的异常，try 块外的异常无法被该 catch 捕获。

  • 多 catch 块的优先级：子类异常的 catch 块需放在父类异常之前（如先 catch NullPointerException，再 catch Exception），否则子类异常会被父类 catch 块覆盖。
  • finally 的作用域：finally 块无论 try 块是否抛出异常、catch 是否捕获，都会执行（除非 JVM 终止），常用于释放资源（如关闭流、连接）。

异常处理后的影响作用域

局部生效，不跨上下文：

  • 方法内处理：catch 块内的修复逻辑仅作用于当前方法的局部变量 / 执行流程，不会影响其他方法的变量或线程。
  • 全局处理（如 Thread.setUncaughtExceptionHandler）：仅作用于当前线程，不同线程的未捕获异常需单独配置处理器，无法跨线程生效。
  • 类级别的异常处理（如 Spring 的 @ControllerAdvice）：作用域覆盖指定包 / 类下的所有控制器方法，超出该范围的非控制器方法异常无法被处理。

String 相关

String 由哪个包定义

核心定义包：java.lang.String，是 Java 核心基础包，无需手动导入。其所有核心属性（如存储字符的 value 数组）、核心方法（如 equals()、substring()）均在此包中实现。

核心关联包：不存在第二个定义 String 类的包，但 java.nio.charset 是 String 功能的核心关联包——String 的编码、解码能力（如 getBytes(Charset)、通过字符集构造 String）完全依赖该包的 Charset 类，因此常被混淆为"第二个定义包"。

Java 标识符命名规范

  • 语法规则：标识符只能由字母（A-Z/a-z）、数字（0-9）、下划线（_）、美元符（$）组成，不能以数字开头，不能使用 Java 关键字（如 int、class）或保留字（如 goto、const），且区分大小写。
  • 编码约定：
    • 包名：全小写多级结构（如 com.example.demo）
    • 类名 / 接口名：大驼峰（如 UserService）
    • 方法名 / 变量名：小驼峰（如 getUserName、orderId）
    • 常量名：全大写 + 下划线分隔（如 MAX_AGE）
    • 枚举类名：大驼峰，枚举值全大写

public class Main {
    /**
     * 找到和为k的倍数的最大和值
     * @param nums 输入数组
     * @param k 目标倍数
     * @return 满足条件的最大和值（无满足条件的子集返回0）
     */
    public static int findMaxSumMultiple(int[] nums, int k) {
        // dp[i]：表示选取某些数后，它们的和对k取模等于i时的最大和值
        int[] dp = new int[k];
        
        // 初始化dp数组：将所有余数状态初始化为Integer.MIN_VALUE，表示初始时不可达
        for (int i = 0; i < k; i++) {
            dp[i] = Integer.MIN_VALUE;
        }
        
        // 初始状态：选取0个数时，和为0，余数为0
        dp[0] = 0;
        
        // 遍历数组中的每一个数字
        for (int num : nums) {
            // 创建dp数组的副本temp，用于存储更新后的状态
            int[] temp = dp.clone();
            
            // 遍历所有可能的余数状态（0到k-1）
            for (int i = 0; i < k; i++) {
                // 如果当前余数状态i是可达的
                if (dp[i] != Integer.MIN_VALUE) {
                    // 计算当前数字num对k取模的余数
                    int remainder = (num % k + k) % k;
                    
                    // 计算加入当前数字后新的余数
                    int newMod = (i + remainder) % k;
                    
                    // 计算加入当前数字后的新和值
                    int newSum = dp[i] + num;
                    
                    // 如果新和值比temp中记录的newMod状态的和值更大，则更新
                    if (newSum > temp[newMod]) {
                        temp[newMod] = newSum;
                    }
                }
            }
            
            // 将更新后的temp数组赋值给dp，完成一轮状态转移
            dp = temp;
        }
        
        // 最终结果：dp[0]存储的是和为k的倍数（余数0）的最大和值
        return dp[0] > 0 ? dp[0] : 0;
    }
    
    public static void main(String[] args) {
        // 测试用例1
        int[] nums1 = {3, 1, 2, 4, 5};
        int k1 = 3;
        System.out.println("测试用例1:");
        System.out.println("数组: [3, 1, 2, 4, 5], k = 3");
        System.out.println("最大和值: " + findMaxSumMultiple(nums1, k1)); // 输出: 15
        System.out.println();
        
        // 测试用例2
        int[] nums2 = {5, 7, 9, 11, 13};
        int k2 = 4;
        System.out.println("测试用例2:");
        System.out.println("数组: [5, 7, 9, 11, 13], k = 4");
        System.out.println("最大和值: " + findMaxSumMultiple(nums2, k2)); // 输出: 40
        System.out.println();
        
        // 测试用例3
        int[] nums3 = {2, 3, 5, 7};
        int k3 = 5;
        System.out.println("测试用例3:");
        System.out.println("数组: [2, 3, 5, 7], k = 5");
        System.out.println("最大和值: " + findMaxSumMultiple(nums3, k3)); // 输出: 15
    }
}

Java 并发编程

ThreadLocal 和 Thread 的关系

ThreadLocal 并非 Thread 类的子类，二者不存在继承关系——ThreadLocal 是独立的工具类，直接继承自 Object。它实现线程隔离的核心逻辑是：每个 Thread 对象内部维护着一个 ThreadLocalMap 成员变量，ThreadLocal 实例作为"键"，通过操作这个与线程绑定的 ThreadLocalMap，让不同线程借助同一个 ThreadLocal 实例，存取各自独立的数据副本，从而实现线程局部变量的隔离效果。

并发编程同步器

Java 并发编程中的同步器是 java.util.concurrent 包下解决线程协作与资源竞争的工具集，既包含专注互斥访问的锁（如 ReentrantLock），也涵盖侧重线程协调的工具（如 CountDownLatch、Semaphore 等）。

import java.util.concurrent.CyclicBarrier;

public class CyclicBarrierExample {
    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(3, () ->
            System.out.println("===== 所有线程到达屏障，执行屏障动作 ====="));
        for (int i = 1; i <= 3; i++) {
            final int threadNum = i;
            new Thread(() -> {
                try {
                    System.out.println("线程" + threadNum + "开始执行任务");
                    Thread.sleep(1000);
                    System.out.println("线程" + threadNum + "到达屏障");
                    barrier.await();
                    System.out.println("线程" + threadNum + "继续执行后续任务");
                } catch (Exception e) {
                    e.printStackTrace();
                }
            }).start();
        }
    }
}

设计原则

里氏代换原则（LSP）

子类必须能够替换掉其父类，且不影响程序的正确性。

遵循里氏代换原则的核心好处：保障程序稳定性与行为可预测性，避免因子类篡改父类契约引发隐蔽 bug；提升父类通用逻辑的复用率；支撑开闭原则，让新增子类时无需修改原有代码，降低系统扩展与维护成本。

Spring Boot 和开发相关

IOC 自动装配方式

  • 按名称装配（byName）：按属性名匹配容器中同名 Bean，XML 配置 autowire="byName"，注解常用 @Resource(name="bean名") 或 @Autowired + @Qualifier，匹配精准，避免同类型 Bean 冲突。
  • 按类型装配（byType）：按属性类型匹配同类型 Bean，XML 配置 autowire="byType"，注解直接用 @Autowired，简化配置，但要求同类型 Bean 唯一，否则需用 @Qualifier 指定名称。
  • 构造器装配（Constructor）：通过构造器注入依赖，XML 配置 autowire="constructor"，注解可省略 @Autowired，优先按参数类型匹配，保证依赖不可变，是 Spring 推荐方式。
  • 方法装配（Method）：通过方法调用注入，含 Setter 方法（@Autowired 标注 setXxx()）、工厂方法、@Bean 方法（Spring Boot 主流）三种形式。

构造器方法用 @Lazy 解决循环依赖，其他三种方式均可通过三级缓存解决循环依赖。

阻塞任务队列的线程池

使用有界阻塞队列：

  1. 线程 A 调用 put()，发现队列满 → 执行 notFull.await()。
  2. 线程 A 释放锁，从"运行状态"变为"等待状态"，退出 CPU 调度（不占资源）。
  3. 线程 B 调用 take() 取出元素，队列空闲 → 执行 notFull.signal().
  4. 线程 A 被唤醒，重新竞争锁 → 拿到锁后重新校验队列是否真的空闲（while 循环）。
  5. 若队列空闲，线程 A 入队元素；若仍满（虚假唤醒），则再次进入 await().

非阻塞入队 offer，拒绝策略进行阻塞入队 put。

IO 相关

流的传递方式

按传输方向

  • 输入流：数据从外部载体（文件、网络、硬件）流向程序 / 内存（如 FileInputStream、BufferedReader）。
  • 输出流：数据从程序 / 内存流向外部载体（如 FileOutputStream、BufferedWriter）。
  • 单个流对象均为单向传输，双向传输需同时创建输入 + 输出流.

按传输粒度

  • 字节流：以 8 位 byte 为单元，适配所有数据（二进制文件、图片、视频、文本），如 InputStream/OutputStream 体系.
  • 字符流：以 16 位 char 为单元，专为文本优化（自动处理编码），是字节流 + 编码转换的封装，如 Reader/Writer 体系.

按传输模式

  • 同步流：调用读写方法后线程阻塞，直到传输完成（默认方式，如 FileReader.read()），逻辑简单但高并发易卡顿.
  • 异步流：调用后线程不阻塞，通过回调 / 事件接收结果（如 Java NIO 的 AsynchronousFileChannel），适配大文件 / 高并发场景.

按传输载体

  • 文件流：磁盘与内存间传输（如 FileInputStream）。
  • 网络流：基于 Socket 的网络节点间传输（TCP 为有序字节流，UDP 为无连接数据报流）。
  • 内存流：内存缓冲区内部传输（如 ByteArrayInputStream），无 IO 开销.
  • 管道流：进程 / 线程间通信（如 PipedReader）。
  • 压缩流：边传输边压缩 / 解压（如 GZIPInputStream）。

按优化方式

  • 非缓冲流：直接读写底层载体，每次操作触发 IO，效率低（如 FileReader）。
  • 缓冲流：先批量读取到内存缓冲区，减少 IO 次数（如 BufferedReader），是最常用的优化方式.

通用规则

  • 顺序性：多数流为"顺序流"，按写入 / 读取顺序传输，随机访问需特殊实现（如 RandomAccessFile）。
  • 资源释放：流关联底层资源（文件句柄、网络连接），需手动 / 自动（try-with-resources）关闭，避免泄漏.
  • 关联性：字符流是字节流的封装，缓冲流是普通流的包装，可多层嵌套优化传输（如 BufferedReader 包装 FileReader）。

字节流和文本文件

特性 字节流（InputStream/OutputStream） 字符流（Reader/Writer）
处理单位 8 位 byte 16 位 char
适用场景 所有文件（图片、视频、二进制、文本） 仅文本文件
编码处理 不处理，逐字节读写 自动处理字符编码转换
读取文本 可以，但需自行处理编码 可以，自动处理编码

两者都可以读取文本文件，但字符流更适合文本处理（自动处理编码，支持按行读取）。

符号链接和硬链接

  • 硬链接：多个文件名直接指向同一个文件本体（inode + 数据块）。系统里的文件本体是通过 inode 标识的，文件名只是给用户看的"访问入口"。创建硬链接，就是新增一个"入口"（文件名），和原文件名共用同一个文件本体。这些文件名之间没有主次之分，删除其中一个，只是少了一个入口，文件本体只要还有其他入口（链接计数 > 0）就不会消失.
  • 符号链接：一个文件名指向另一个文件名。符号链接本身是一个独立的小文件，里面只存了目标文件的路径字符串。当你访问符号链接时，系统会先根据这个路径找到目标文件名，再通过目标文件名找到文件本体。这就是为什么删除目标文件后，符号链接会变成"悬空链接"——它指向的那个目标文件名没了，自然找不到文件本体.

public static boolean isValidFileName(String fileName) {
    // 1. 空值校验，.trim() 去除字符串首尾的所有空白字符
    if (fileName == null || fileName.trim().isEmpty()) {
        return false;
    }
    // 2. 按下划线分割，检查分割后的部分数量是否为 3
    // 若不做空值校验，fileName 为 null 时，这行直接抛异常
    String[] parts = fileName.split("_");
    if (parts.length != 3) {
        return false;
    }
    // 3. 校验 PROJECT 部分是否为 MSD358
    if (!"MSD358".equals(parts[0])) {
        return false;
    }
    // 4. 校验 VERSION1 和 VERSION2 是否为 6 位纯数字
    // ^ 匹配开头，$ 匹配结尾，\d 数字，{6} 恰好六位
    String sixDigitPattern = "^\\d{6}$";
    if (!parts[1].matches(sixDigitPattern) || !parts[2].matches(sixDigitPattern)) {
        return false;
    }
    // 5. 转换为长整型并校验 VERSION1 < VERSION2
    long version1;
    long version2;
    try {
        version1 = Long.parseLong(parts[1]);
        version2 = Long.parseLong(parts[2]);
    } catch (NumberFormatException e) {
        return false;
    }
    return version1 < version2;
}

回溯空间树

回溯法通过构建解空间树来系统地搜索所有可能解：

  • 根节点：初始状态（空解）。
  • 分支：每层节点代表一次选择（如选取或不选取某元素）。
  • 叶子节点：完整解或无效解。
  • 剪枝：当某分支不可能产生有效解时，提前终止搜索（剪去该子树），避免无效遍历。

回溯法的核心思想：递归深入 → 判断是否有效 → 无效则回退（回溯）到上一层，尝试下一个分支。

public static long findMinBitwiseAnd(long[] nums, int k) {
    if (k <= 0 || k > nums.length) {
        return 0;
    }
    long resultMask = 0L;
    // 从最高位到最低位尝试构建掩码
    for (int bit = 60; bit >= 0; bit--) {
        long candidateMask = resultMask | (1L << bit);
        int count = 0;
        for (long num : nums) {
            if ((num & candidateMask) == candidateMask) {
                count++;
            }
        }
        if (count >= k) {
            resultMask = candidateMask;
        }
    }
    // 所有不在 resultMask 中的位都置 1
    long minResult = 0L;
    for (int bit = 60; bit >= 0; bit--) {
        if ((resultMask & (1L << bit)) == 0) {
            minResult |= (1L << bit);
        }
    }
    return minResult;
}

算法思路：贪心策略，从高位到低位依次判断是否可以将该位设为0，同时保证至少有k个数满足条件。

回溯：子集树问题

回溯空间树：回溯算法通过构建状态空间树来搜索解，每个节点代表问题的一个状态，从根节点到叶子节点的路径构成一个完整的解。

核心思想：

  • 深度优先搜索：沿着一条路径深入探索直到无法继续
  • 剪枝优化：通过约束函数和限界函数提前剪除无效分支
  • 回溯：当当前路径无法得到解时，返回上一层尝试其他选择

应用场景：排列组合、子集和、N皇后、图着色等组合优化问题。

        System.out.println("Math.ceil(5.0) = " + Math.ceil(5.0));       // 5.0
    }
}
  • Math.floor()：向下取整（趋向负无穷）。
  • Math.ceil()：向上取整（趋向正无穷）。

Bug 的生命周期

  1. 新建（New）：测试人员在缺陷管理工具（如 Jira）中提交 Bug 单，记录复现步骤、环境、预期与实际结果等信息。
  2. 确认（Confirmed）：Bug 负责人审核验证，确认是否为真实存在的缺陷；若为无效 Bug 则直接关闭。
  3. 分配（Assigned）：将有效 Bug 指派给对应的开发人员，标注优先级和严重程度。
  4. 修复中（In Progress）：开发人员定位问题根因并编写修复代码。
  5. 已修复（Fixed）：开发人员完成修复、自测通过后，将 Bug 回传给测试人员。
  6. 回归测试（Retest）：测试人员验证 Bug 是否真正解决；若未修复则打回"重新打开（Reopened）"。
  7. 关闭（Closed）：回归测试通过后，Bug 归档关闭。

算法

时间复杂度

T(n) = 2T(n/2) + n

根据主定理（Master Theorem），a = 2，b = 2，f(n) = n，log_b(a) = 1，属于 Case 2，因此 O(n log n).

给n个数，取部分数的和为k的倍数，要求数最大

算法思路：使用动态规划，dp[i] 表示和为 k 的倍数中余数为 i 的最大和值。

核心步骤：

  1. 初始化 dp 数组，dp[0] = 0，其余为 Integer.MIN_VALUE
  2. 遍历每个数字，创建 dp 副本避免重复使用
  3. 更新每个余数状态的最大和值
  4. 返回 dp[0] 即为答案

时间复杂度：O(n×k)，空间复杂度：O(k)
