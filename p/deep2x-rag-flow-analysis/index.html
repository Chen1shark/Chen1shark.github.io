deep2X RAG 流程完整解析

从文档分块、向量化、消息队列、检索召回、Prompt 组装、流式输出、事务一致性到 Redis 使用，完整说明 deep2X 项目中的 RAG 端到端实现。

2026-04-25

文档定位

本文只分析 deep2X 项目中的 RAG 相关流程，不展开无关的系统管理、代码生成、普通 CRUD 页面等内容。RAG 在项目中不是一个单独的方法，而是一条跨服务链路，涉及 knowledge-agent、consumer、agent-rest、PostgreSQL/pgvector、RabbitMQ、Redis、Spring AI、OpenFeign、Nacos、MyBatis-Plus 等组件。

RAG 的核心目标是：用户提问时，不让大模型只依赖自身训练知识回答，而是先从项目知识库中召回与问题相关的文本分段，再把这些分段作为上下文交给大模型生成回答。这样可以让回答基于企业私有文档，减少幻觉，并且能随知识库更新而实时生效。

RAG 总体链路

deep2X 的 RAG 可以拆成两个阶段：

  1. 知识入库阶段：把文档内容拆成多个文本分段，保存成向量化任务，通过 RabbitMQ 异步调用 Embedding 模型，最后把向量写入 cw_text_vector_task。
  2. 检索问答阶段：用户提问后，把问题向量化或分词，按智能体配置执行向量检索、全文检索或混合检索，得到相关分段，再组装历史对话、知识库上下文和当前问题，通过 Spring AI 以 SSE 方式流式输出。

知识入库阶段

文件上传与分块边界

项目把文件解析与文本分块放在导入层完成，导入层负责处理 PDF、Word、Markdown、网页等原始内容，并把清洗后的文本组织成分段列表。后端 RAG 入库边界非常清晰：knowledge-agent 接收已经拆好的分段，统一保存为向量化任务，然后触发异步 Embedding。

前端页面文案中体现了多种导入方式：文档导入、问答对导入、自动拆分问答对、网页解析。后端负责接收这些导入结果，并把每个分段保存为 cw_text_vector_task 中的一行。

这意味着项目中的“文本分块”是一套导入层约定：在调用 /vector/batchSave 前，系统已经把原始文档拆成了多个 TextVectorTaskBo，每个对象至少包含 content，可以附带 sort、answer、png、video、doc 等信息。

为什么要分块

RAG 必须分块，原因不是为了“保存方便”，而是为了检索质量和模型上下文成本：

  • 原始文档通常很长，不能整篇塞进大模型上下文，否则 Token 成本高、延迟大，还会超过模型窗口限制。
  • 向量检索的基本单位越合理，召回越准确。整篇文档向量会稀释主题，导致一个局部问题找不到对应段落。
  • 分块后可以只召回 Top-K 相关片段，把无关内容排除在 Prompt 外，提高回答聚焦度。
  • 分块可以保留 sort，后续需要按原文顺序拼接上下文时可以恢复局部顺序。
  • 分块可以支持 QA 格式数据。若 answer 不为空，系统可以把“问题-答案”对直接注入 Prompt，而不是只注入普通文本。

分块策略

deep2X 采用“导入层分块，业务层入库”的方式。导入层按照不同数据类型选择不同分块策略：

数据类型 分块方式 为什么这样分
普通长文档 按标题、自然段、标点边界切分，再控制最大长度 保留语义完整性，避免句子被截断
产品手册或制度文件 优先按章节、条款、列表项切分 用户常问具体规则，条款级分段更容易命中
问答对数据 一条问题和答案作为一个分段 检索到问题后可以把答案直接给模型作为高置信上下文
表格类文本 按行或业务对象切分，并保留表头 避免一行数据脱离字段名后不可理解
网页解析内容 去除导航、广告、页脚后按正文段落切分 降低噪声进入知识库的概率

分块时不只按固定字符数硬切。固定长度切分虽然实现简单，但容易把定义、条件、结论切开，导致检索命中的分段缺少上下文。deep2X 采用“语义边界优先，长度限制兜底”的方式，先保留章节、段落、条款等自然边界，再用长度限制控制单段大小。

分块长度与重叠

项目会控制每个分段在适合 Embedding 和 Prompt 注入的范围内，并在导入层处理必要的分段重叠。分段长度与重叠的取舍如下：

  • 单段过短：向量语义不足，容易只匹配到孤立关键词。
  • 单段过长：向量会混合多个主题，检索准确率下降。
  • 适当重叠：可以避免边界处的信息丢失，例如上一段结尾是问题条件，下一段开头是结论。
  • 重叠不宜过大：过大会造成重复内容太多，浪费存储和 Embedding 成本，也会让召回结果高度重复。

项目中 contentLength 会在保存时写入，来自 bo.getContent().length()。这个字段可用于页面统计、文件训练进度展示，也可以作为后续质量分析指标，例如发现某个文件分段过短或过长。

@Transactional
@Override
public Boolean batchSave(TextVectorTaskDto textVectorTaskDto) throws Exception {
    List<TextVectorTaskBo> bos = textVectorTaskDto.getTextVectorTaskBos();
    bos.forEach(bo -> {
        bo.setDocName(textVectorTaskDto.getDocName());
        bo.setFilePath(textVectorTaskDto.getFilePath());
        bo.setFileId(textVectorTaskDto.getFileId());
        bo.setKnowId(textVectorTaskDto.getKnowId());
        bo.setKnowName(textVectorTaskDto.getKnowName());
        bo.setModelId(textVectorTaskDto.getModelId());
    });
    for (TextVectorTaskBo bo : bos) {
        this.save(bo);
    }
    return true;
}

这里为什么不是 saveBatch 一次性写库？因为 save(bo) 不只是插入数据库，它还会在每条分段保存后投递对应的向量化消息。如果直接 saveBatch，需要额外再遍历任务 ID 投递 MQ。当前写法把“保存分段”和“发送该分段向量化任务”绑定在一起，逻辑更直观。

但这种写法也有一个要注意的点：batchSave 中使用 this.save(bo) 是同类内部调用，save(bo) 方法上的 Spring AOP 事务代理不会再次生效。不过外层 batchSave 本身有 @Transactional，所以批量保存仍然处在外层事务中。

这里先写数据库再发 MQ，是为了确保 MQ 消息中的 taskId 在消费者处理时一定能查到。消息内容只带任务 ID，不带完整文本，因此消费者必须回查 cw_text_vector_task。如果先发消息再写数据库，消费者可能抢先消费，导致查不到任务。

消息队列与异步向量化

为什么使用 RabbitMQ

Embedding 向量化是高延迟、易失败、受外部模型服务影响的操作。如果在 /vector/batchSave 请求中同步完成所有分段的 Embedding，会带来几个问题：

  • 用户上传大文件时，请求会长时间阻塞，前端体验差。
  • Embedding API 可能限流、超时或波动，同步处理会直接影响主业务接口可用性。
  • 批量文档入库可能瞬间产生大量模型调用，需要队列削峰。
  • 向量化失败需要重试和死信记录，如果没有 MQ，会把重试逻辑写散在业务接口里。
  • 消费者可以独立扩容，提升向量化吞吐，不需要扩容整个管理服务。

因此项目把“保存分段”和“生成向量”解耦：knowledge-agent 只负责创建任务和发送消息，consumer 负责慢任务处理。

消息持久化如何保证不丢

RabbitMQ 要做到“尽量不丢消息”，不能只声明 durable 队列，需要同时满足四层条件：

层级 做法 解决的问题
Exchange 持久化 new DirectExchange(EXCHANGE_NAME) 默认 durable 为 true Broker 重启后交换机不丢
Queue 持久化 QueueBuilder.durable(QUEUE_NAME) Broker 重启后队列不丢
Message 持久化 消息 deliveryMode=PERSISTENT Broker 重启后已入队消息不丢
Confirm/Return 开启生产者确认和路由失败回调 生产端知道消息是否进入 Broker 或是否路由失败

@RabbitListener(queues = RabbitMQConfig.QUEUE_NAME)
public void handleMessage(MessageDto messageDto, Channel channel,
                          @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
    try {
        messageService.processMessage(messageDto);
        channel.basicAck(tag, false);
    } catch (Exception e) {
        int retryCount = messageDto.getRetryCount() == null ? 0 : messageDto.getRetryCount();
        if (retryCount < 3) {
            messageDto.setRetryCount(retryCount + 1);
            rabbitTemplate.convertAndSend(RabbitMQConfig.EXCHANGE_NAME, RabbitMQConfig.ROUTING_KEY, messageDto);
            channel.basicAck(tag, false);
        } else {
            channel.basicNack(tag, false, false);
        }
    }
}

这里的关键点是：

  • 成功处理后 basicAck，表示这条消息可以从队列删除。
  • 可重试失败时，先投递一条带新 retryCount 的消息，再 ACK 原消息。
  • 超过重试次数后 basicNack(tag, false, false)，表示拒绝当前消息且不回到主队列，由队列上的死信配置接管。
  • prefetch: 1 可以避免一个消费者一次拿太多消息，减少某个消费者卡住时造成的消息堆积和处理不均。

死信队列作用

死信队列不是为了“自动修复问题”，而是为了不让失败任务沉默丢失。Embedding 任务失败可能来自多种原因：

  • 模型厂商密钥错误。
  • 模型名配置错误。
  • 文本过长或格式异常。
  • 外部 API 超时。
  • pgvector 维度与返回向量维度不一致。
  • 数据库连接异常。

死信队列保存这些失败任务后，运维或后台任务可以按 messageId、content、retryCount 定位问题，并决定是人工修复配置后重投，还是标记任务失败。

public static List<Double> normalizeVectorWithND4J(List<Double> vector) {
    if (vector == null || vector.isEmpty()) {
        return vector;
    }

    double[] array = vector.stream().mapToDouble(Double::doubleValue).toArray();
    INDArray ndArray = Nd4j.create(array);
    double norm = ndArray.norm2Number().doubleValue();

    if (Math.abs(norm) < 1e-12) {
        return vector.stream().map(d -> 0.0).collect(Collectors.toList());
    }

    INDArray normalized = ndArray.div(norm);
    return Arrays.stream(normalized.toDoubleVector()).boxed().collect(Collectors.toList());
}

L2 归一化就是把向量缩放到长度为 1。这样做的原因是：

  • 余弦相似度关注方向，不关注向量长度，更适合语义相似度。
  • 不同输入文本长度可能导致向量模长差异，归一化可以降低长度影响。
  • 归一化后，内积和余弦相似度具有更稳定的关系。
  • 检索阈值更容易解释，例如 0.6、0.8 表示相似度门槛。

代码中对零向量做了保护，如果模长接近 0，就返回同维度全零向量，避免除以 0。

private String hanlpSegmentToTsQuery(String text) {
    if (text.length() < 4) {
        return text;
    }
    String replaceStr = " & ";
    if (text.length() > 10) {
        replaceStr = " | ";
    }
    return HanLP.segment(text)
            .stream()
            .map(term -> term.word)
            .filter(word -> word.length() > 1)
            .collect(Collectors.joining(replaceStr));
}

这个方法做了几件事：

  • 短文本直接返回原文本，避免过度分词。
  • 文本长度不超过 10 时，用 & 拼接，要求多个词同时命中，提升精确度。
  • 文本长度超过 10 时，用 | 拼接，允许任一关键词命中，提升召回率。
  • 过滤长度为 1 的词，减少“的”“了”“是”等单字噪声。

把知识库内容放进 SystemMessage，是为了让模型把它当成更高优先级的上下文，而不是普通用户补充信息。这样可以减少模型忽略知识库内容的概率。

  • answer 为空：当成普通文本分段，只拼接 content。
  • answer 不为空：当成 QA 数据，拼成“问题：…，答案：…”。

QA 数据的优势是答案更明确。比如知识库里本来就是“问题-答案”格式，模型不需要再从长文本中推理答案，只需要根据召回的 QA 对组织语言。

历史对话的作用

历史对话以 UserMessage 和 AssistantMessage 交替加入。这样用户可以继续追问，例如：

  • 第一轮：“这个产品的保修期多久？”
  • 第二轮：“那人为损坏呢？”

第二轮问题本身没有明确主语，如果没有历史对话，模型不知道“那”指什么产品。加入历史记录后，模型能保持上下文连续性。

上下文数量控制

项目按智能体配置控制历史上下文轮数，普通聊天接口默认使用最近 10 条记录。这样可以在保证多轮对话连续性的同时，避免历史过多导致以下问题：

  • Prompt 变长，成本上升。
  • 早期无关对话干扰当前回答。
  • 超过模型上下文窗口后请求失败。

因此项目优先保留最近对话，并通过 contextNum 控制上下文规模。

SSE 的工作方式

SSE 是服务端到客户端的单向事件流。后端每收到模型的一段输出，就把这段内容推给前端。前端不需要等完整响应结束，就可以实时渲染。

在本项目中：

  • 后端流：Flux<ChatResponse>。
  • 模型流：StreamingChatModel.stream(prompt)。
  • 响应类型：text/event-stream。
  • 累积完整答案：fullResponse.append(content)。
  • 完整保存：流结束后把 fullResponse 写入对话记录。

为什么需要累积完整回答

流式输出只解决前端实时展示，不等于数据库已经保存了完整答案。数据库需要保存完整问答记录，因此后端必须在流式过程中同步累积内容。

如果不累积完整回答，只把每个 token 发给前端，流结束后服务端就不知道完整答案是什么，也无法保存历史记录。

deep2X 使用 doFinally 保存流式回答。doOnComplete 只在流正常完成时触发，如果客户端中途关闭页面、网络断开或取消订阅，Reactor 触发的是 cancel，不是 complete，因此不能只依赖 doOnComplete。doFinally 能覆盖正常完成、异常和取消三类终止状态：

普通同步业务为什么可以用 @Transactional

流式回调为什么使用显式事务

SSE 流式链路不完全等同于普通同步方法。保存记录发生在 Reactor 回调中，例如 doOnComplete 或 doFinally。这类回调可能运行在响应式链路的线程中，而不是进入 Controller 时的原始线程。

因此，在流式结束回调里，如果要同时做多步数据库操作，例如：

  • 首次对话时更新会话标题。
  • 插入对话记录。
  • 保存引用分段。
  • 更新会话最后对话时间。
  • 更新使用次数或配额。

deep2X 在流式结束保存中使用 TransactionTemplate 显式事务，而不是只依赖方法上的 @Transactional。原因是显式事务会在回调执行线程里主动开启、提交或回滚事务，事务边界更清楚，不依赖响应式回调能否正确走到代理方法。

写法如下：

为什么不是所有地方都改成显式事务

显式事务更啰嗦，不适合所有方法。项目里可以按场景区分：

场景 使用方式 原因
普通同步 CRUD @Transactional 简洁，Spring AOP 能稳定管理
批量导入保存 @Transactional 外层事务足够清晰
SSE 结束回调保存 TransactionTemplate 回调线程和生命周期特殊，显式事务更可控
MQ 消费单条任务 视情况使用显式事务 消息 ACK 和数据库提交需要明确顺序

因此，“不用注解而是显式写事务”的准确含义不是项目完全不用 @Transactional，而是在流式回调和消息消费这类异步边界里，显式事务更适合保证数据一致性。

Redis 缓存一致性策略

deep2X 采用 Cache-Aside 策略：

  1. 读数据时先查 Redis。
  2. Redis 命中则直接返回。
  3. Redis 未命中则查数据库。
  4. 数据库返回后写入 Redis，并设置 TTL。
  5. 写操作成功后删除相关缓存。

这种策略的优点是简单、稳定、适合读多写少的数据。智能体配置、模型列表、网站配置、检索短缓存都适合用这种模式。

Redis 缓存不能替代数据库。RAG 的最终事实数据仍然以 PostgreSQL 和 MySQL 为准，Redis 只承担加速、临时状态和分布式会话职责。

关键实现细节

分块职责放在导入层

文件解析、内容清洗、标题识别、段落切分、QA 抽取等工作在导入层完成，knowledge-agent 只接收标准化后的分段列表。这种方式让后端 RAG 主链路保持稳定：无论来源是 PDF、Word、Markdown、网页还是 Excel 问答对，最后都统一转换为 TextVectorTaskBo。

这样做的优势是职责清晰。导入层关注“如何把不同格式的原始内容变成干净分段”，RAG 入库层关注“如何保存分段、投递任务、生成向量”。当后续新增网页解析、OCR 或更复杂的文档清洗策略时，不需要改动向量化和检索主流程。

MQ 使用手动 ACK 和死信队列

RabbitMQ 承接 Embedding 任务，并通过持久化队列、持久化消息、生产者 Confirm、ReturnCallback、消费者手动 ACK 和死信队列保证消息可靠性。

手动 ACK 的核心价值是把“业务处理成功”和“消息确认删除”绑定在一起。只有向量化成功并写回数据库后，消费者才执行 basicAck。如果处理失败，系统根据 retryCount 重新投递；超过最大重试次数后执行 basicNack(tag, false, false)，消息进入死信队列。

这种方式比默认自动 ACK 更可靠，因为默认自动 ACK 容易出现“消费者拿到消息后业务失败，但消息已经被确认”的情况。RAG 入库任务一旦丢失，就会造成分段一直没有向量，后续检索永远召回不到该知识。

流式保存使用 doFinally

SSE 对话保存使用 doFinally，而不是只使用 doOnComplete。原因是 SSE 有三种结束状态：正常完成、异常结束、客户端取消订阅。

doOnComplete 只覆盖正常完成。用户关闭页面、浏览器刷新、移动端网络切换时，服务端通常收到的是 cancel 信号，不会触发 doOnComplete。doFinally 能统一覆盖 ON_COMPLETE、ON_ERROR、CANCEL，因此更适合保存流式回答。

流式回调使用显式事务

流式结束保存记录时使用 TransactionTemplate。这是因为流式回调发生在 Reactor 生命周期中，和普通同步 Controller 方法不同。显式事务可以在回调执行线程中主动开启事务，把更新会话标题、插入对话记录、保存引用分段、更新会话时间等操作放进同一个事务边界。

显式事务的优势是边界清楚、失败可控。任意一步写库失败，都可以通过 status.setRollbackOnly() 回滚前面已经执行的数据库操作，避免出现会话标题已更新但对话记录没插入，或记录已插入但引用分段没保存的问题。

流式文本累积使用 StringBuilder

SSE 单次请求内使用 StringBuilder 累积模型输出。这里选择 StringBuilder，是因为每一次对话请求都会创建独立的局部变量，变量不在不同用户请求之间共享；同时 Spring AI 的单个流式订阅按顺序回调 doOnNext，没有把同一个 fullResponse 放到多个线程并行写入。

StringBuilder 比 StringBuffer 更适合这个场景，因为 StringBuffer 的 append 方法带同步锁，而当前累积过程不需要跨线程共享。没有并发写入时，StringBuffer 的锁只会增加不必要的开销。

真正需要保证的是“不要把同一个 StringBuilder 提升成 Bean 字段或静态变量”。只要它保持在方法局部作用域，每个 SSE 请求独占一个实例，就不会出现不同用户回答串写的问题。

多知识库结果全局重排

智能体可以关联多个知识库。项目不会简单地把每个知识库的结果直接拼接进 Prompt，而是把多个知识库召回结果合并后按 similarity 倒序排列，再用 limit(callbackNumber) 截断。

这样做可以避免某个知识库因为遍历顺序靠前而占满上下文。最终进入 Prompt 的是全局最相关的 Top-K 分段，而不是每个知识库各自返回的局部 Top-K。

RAG 服务使用 PostgreSQL 数据源

RAG 相关查询通过 @DS("pgsql") 路由到 PostgreSQL。这个选择由能力决定：向量检索依赖 pgvector 的 <=> 操作符，全文检索依赖 to_tsvector、to_tsquery 和 ts_rank，这些都属于 PostgreSQL 能力。

系统管理数据仍然走 MySQL 的 system 数据源，RAG 知识库、向量、智能体配置和对话记录走 PostgreSQL 的 pgsql 数据源。这样既保留了传统管理系统的数据结构，又让 RAG 检索获得 pgvector 支持。

pgvector 使用余弦距离

向量检索以 <=> 为主，它表示 cosine distance，值越小越相似。为了让业务层更容易理解，SQL 中使用 1 - (embedding <=> query) 转成相似度分数，值越大越相关。

这种写法和页面上的“相似度阈值”语义一致。比如阈值设置为 0.6，就表示只保留相似度大于 0.6 的分段，而不是让业务方理解“距离小于多少”。

总结

deep2X 的 RAG 本质是一条异步知识入库与实时检索生成链路。入库阶段通过分段任务、RabbitMQ、Embedding 模型和 pgvector 建立可检索的知识索引；问答阶段通过智能体配置选择检索策略，把召回分段、历史对话和当前问题一起交给大模型生成回答。

这套设计的关键点不是单个 API，而是多个工程取舍的组合：文本必须合理分块，Embedding 必须异步化，MQ 必须有可靠投递和死信兜底，检索必须支持向量、全文和混合模式，SSE 必须处理取消和异常，事务必须在异步边界显式可控，Redis 则负责登录态、热点配置、短期检索结果、流式状态和限流配额。
