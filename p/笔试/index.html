操作系统

文件逻辑结构分类

无结构文件（流式文件）

定义：

  • 文件被视为一个整体，不进行任何内部划分
  • 以字节为单位的有序序列，无固定记录单位
  • 操作系统只关注文件的存取，不解释文件内容

特点：

  • 数据组织完全由应用程序决定
  • 灵活性最高，适用于任意类型数据
  • 读写以字节或字符流形式进行

常见示例：

文件类型 说明
文本文件（.txt） 纯字符序列，无格式记录划分
源代码文件（.java, .c） 代码文本的连续字符流
二进制可执行文件 机器指令的连续字节序列
日志文件 按行追加的文本内容

有结构文件（记录式文件）

定义：

  • 文件由若干**记录（Record）**组成
  • 每条记录有固定格式和字段定义
  • 记录是文件的基本操作单位（最小读写单元）

特点：

  • 支持按记录号或关键字进行存取
  • 适用于数据库、表格化数据管理
  • 系统需理解记录格式

常见示例：

场景 记录结构
学生成绩文件 学号 | 姓名 | 语文 | 数学 | 英语
职工档案 工号 | 部门 | 职位 | 入职日期 | 薪资
银行账户 账号 | 户名 | 余额 | 开户行 | 状态
有结构文件的四种组织方式

1. 顺序文件

定义：记录按某个关键字（如学号、工号）顺序排列存储。

特点：

  • 物理顺序与逻辑顺序一致
  • 支持顺序存取（从第一个记录依次读取）
  • 查找特定记录需从头遍历，效率较低

示例：学生成绩按学号从小到大排列

记录3: 2021005 | 王五 | 90 | 85 | 88

适用场景：批处理任务、报表生成、顺序遍历全量数据

2. 索引文件

定义：为文件建立索引表，索引项包含关键字和对应记录的物理地址。

特点：

  • 索引表与数据文件分离存储
  • 支持直接存取（通过索引快速定位）
  • 插入删除灵活，只需修改索引

  • 查找时先查索引定位组，再组内顺序查找
  • 平衡了索引空间和查询效率

为什么没有"连续文件"作为主要类型

概念澄清：逻辑结构 vs 物理结构
维度 逻辑结构 物理结构
关注点 用户看到的数据组织方式 数据在存储设备上的实际存放
视角 应用程序/用户视角 操作系统/磁盘视角
典型例子 流式文件、索引文件 连续分配、链接分配、索引分配

关键理解：

  • 连续分配是物理存储方式，指文件数据在磁盘上占用连续的物理块
  • 逻辑结构讨论的是"文件内容如何组织"，而非"数据放在磁盘哪里"
连续分配的物理结构缺点

1. 外部碎片严重

  • 磁盘上产生大量无法利用的小空闲区
  • 需定期执行磁盘整理（碎片整理）来合并空闲空间

2. 文件增长受限

  • 创建时必须声明最大文件大小
  • 若实际增长超出预分配空间，需整体迁移

3. 插入删除代价高

  • 在文件中间插入记录需移动后续所有数据
  • 删除记录后需压缩空间或标记空闲

4. 存储利用率低

  • 预分配空间往往大于实际使用，造成浪费
现代文件系统的物理结构选择

现代操作系统已放弃连续分配作为主方案，采用：

系统 物理结构 说明
Unix/Linux 索引分配（inode） 多级索引，支持大文件和高效随机访问
Windows NTFS B+树索引 主文件表（MFT）记录文件属性及数据位置
现代数据库 页式管理+索引 按页存储，通过 B+树索引组织
总结：为什么考试会问这个问题

操作系统教材中"文件逻辑结构"只讲：

  • 无结构文件（流式）
  • 有结构文件（记录式及其变种）

而"连续分配"出现在"文件的物理结构"章节，与"链接分配"、“索引分配"并列。混淆两者是常见考点，需明确区分：

逻辑结构 = 数据的逻辑组织（是什么） 物理结构 = 数据的物理存储（放哪里）

虚拟内存

什么是虚拟内存

定义：

虚拟内存（Virtual Memory）是一种内存管理技术，它为每个进程提供一个连续的、私有的地址空间假象，使得进程以为自己独占整个内存，而实际上物理内存可能被多个进程共享，且部分数据可能存储在磁盘上。

核心思想：

  • 地址分离：程序使用虚拟地址（逻辑地址），CPU 通过地址转换映射到物理地址
  • 空间扩展：利用磁盘空间作为内存的后备存储，突破物理内存容量限制
  • 隔离保护：每个进程拥有独立的虚拟地址空间，互不干扰

为什么需要虚拟内存：

问题 虚拟内存解决方案
物理内存不足 将不常用数据换出到磁盘（Swap/Pagefile）
内存碎片 虚拟地址连续，物理地址可离散分布
进程间干扰 每个进程独立地址空间，无法直接访问其他进程内存
程序移植困难 程序使用虚拟地址，与物理内存布局无关

是什么实现了虚拟内存

虚拟内存的实现依赖硬件 + 操作系统的协同工作：

1. 硬件支持：MMU（内存管理单元）

**MMU（Memory Management Unit）**是实现虚拟内存的核心硬件组件，集成在 CPU 内部或紧耦合于 CPU。

MMU 的主要功能：

功能 说明
地址转换 将虚拟地址转换为物理地址
页表遍历 自动查询页表完成地址映射
TLB 缓存 缓存近期地址转换结果，加速后续访问
权限检查 检查页面访问权限（读/写/执行），违规触发缺页异常

虚拟内存 vs 物理内存 vs 磁盘

对比项 虚拟内存 物理内存（RAM） 磁盘（Swap）
容量 理论上无限（受地址位数限制，如 64 位系统为 2^64） 实际硬件容量（如 16GB） 磁盘分区大小（如 32GB）
速度 无实际存储，需地址转换 纳秒级（ns） 毫秒级（ms），慢 10^5 倍
成本 无额外硬件成本 较贵（$/GB 高） 便宜（$/GB 低）
易失性 概念层，不涉及 断电丢失 断电保留
管理方 OS + MMU 硬件 OS + 磁盘驱动

常见笔试考点

考点 1：页面置换的 Belady 异常

FIFO 算法可能出现：分配页框数增加，缺页次数反而增加的异常现象。LRU 和 Clock 算法不会出现。

考点 2：虚拟地址空间大小

32 位系统：2^32 = 4GB 虚拟地址空间 64 位系统：2^64（实际只用 48-57 位，约 256TB-128PB）

考点 3：抖动（Thrashing）

如果进程工作集大小超过物理内存，频繁换入换出，CPU 利用率急剧下降。解决方案：降低并发度、增加物理内存、优化局部性。

考点 4：写时复制（Copy-On-Write）

fork() 创建子进程时，父子共享物理页面，标记为只读。任一进程写入时触发缺页，OS 复制页面后再写入，减少不必要的内存复制。

进程的基本状态

三种基本状态

操作系统中，进程在其生命周期内主要处于以下三种基本状态：

状态 定义 触发条件 特点
就绪（Ready） 进程已具备运行条件，等待 CPU 调度 进程创建完成、I/O 完成、时间片到 在就绪队列中排队，一旦获得 CPU 即可执行
运行（Running） 进程正在 CPU 上执行指令 被调度程序选中，获得 CPU 使用权 单核系统只有一个进程处于运行态；多核可有多个
阻塞/等待（Blocked/Waiting） 进程因等待某事件而暂停执行 请求 I/O、等待信号量、等待用户输入 即使 CPU 空闲也无法运行，需等待事件完成后转为就绪

状态转换关系：

转换 源状态 目标状态 触发原因
1 创建 就绪 进程创建完成，初始化完毕
2 就绪 运行 被调度程序选中，获得 CPU
3 运行 就绪 时间片用完、或被更高优先级进程抢占
4 运行 阻塞 请求 I/O 操作、申请资源失败、等待信号/事件
5 阻塞 就绪 I/O 完成、资源可用、信号到达
6 运行 终止 进程执行完成、异常退出、被杀死

状态转换详解：

  1. 就绪 → 运行：进程调度程序从就绪队列选择一个进程，分配 CPU
  2. 运行 → 就绪：时间片用完、或被更高优先级进程抢占
  3. 运行 → 阻塞：进程执行 I/O 操作（如读写文件）、申请资源失败、等待信号
  4. 阻塞 → 就绪：I/O 完成、资源可用、信号到达，进程重新具备运行条件

为什么没有"挂起"作为基本状态

概念澄清：基本状态 vs 扩展状态
类型 含义 包含状态
基本状态 描述进程对 CPU 和 I/O 的需求关系 就绪、运行、阻塞
扩展状态 描述进程的内存驻留情况（与换入换出相关） 挂起就绪、挂起阻塞

核心区别：

  • 基本状态关注”进程能否使用 CPU"
  • 挂起状态关注"进程是否在内存中"（是物理内存管理范畴）
挂起（Suspend）的本质

挂起不是独立的基本状态，而是对现有状态的修饰/扩展：

1

3
4

就绪状态  ──→  挂起就绪（就绪 + 在外存）
                （进程就绪但映像在磁盘）

阻塞状态  ──→  挂起阻塞（阻塞 + 在外存）
                （进程阻塞且映像在磁盘）
为什么教材不把挂起列为基本状态

1. 分类维度不同

维度 基本状态分类 挂起分类
关注点 进程对 CPU 的需求 进程在内存还是外存
解决的问题 CPU 调度 内存空间管理
历史背景 所有操作系统都必须有 仅在支持虚拟内存/交换的系统中有

2. 早期操作系统没有挂起

  • 早期批处理系统、简单分时系统不支持进程换出
  • 只有具备**交换（Swapping）**功能的系统才需要挂起概念
  • 挂起是现代虚拟内存系统的扩展，非基础功能

挂起与其他概念的区分
概念 含义 与挂起的区别
阻塞 等待 I/O 或事件，进程在内存中 挂起与阻塞正交，可同时存在
挂起 进程映像被换出到磁盘 阻塞进程可被挂起，就绪进程也可被挂起
睡眠 用户态的主动等待（如 sleep()） 实现层面，通常对应阻塞态
僵尸（Zombie） 进程已终止但父进程未回收 特殊的终止态，与挂起无关
典型笔试问法与解答

问：进程有哪些基本状态？

答：三种基本状态——就绪、运行、阻塞。有些系统扩展为五态（新增新建、终止）或七态（新增两种挂起状态），但核心仍是就绪、运行、阻塞三态的变体。

问：进程从运行态可以直接转换到哪些状态？

答：

  • 运行 → 就绪（时间片用完、被抢占）
  • 运行 → 阻塞（请求 I/O、等待资源）
  • 运行 → 终止（执行完成、异常退出）
  • 运行 → 挂起就绪（被换出，较少见）

问：就绪和挂起就绪的区别？

答：

  • 就绪：进程具备运行条件，进程控制块(PCB)和程序都在内存
  • 挂起就绪：进程具备运行条件，但程序映像在外存磁盘，需要先激活（换入内存）才能真正运行。

树的遍历算法

遍历的分类体系

分类维度 类型 说明
搜索策略 BFS（广度优先） 按层次遍历，先访问根，再访问子节点
DFS（深度优先） 沿着一条路径走到底，再回溯
DFS 的访问顺序 先序遍历 根 → 左子树 → 右子树
中序遍历 左子树 → 根 → 右子树
后序遍历 左子树 → 右子树 → 根

BFS（广度优先搜索）

核心思想：

  • 按层次遍历，从上到下、从左到右
  • 使用队列（Queue）实现

遍历过程：

  1. 将根节点入队
  2. 取出队首节点，访问该节点
  3. 将该节点的左子节点、右子节点依次入队
  4. 重复步骤 2-3，直到队列为空

示例树结构：

1
2
3
4
5

        1          ← 第1层
       / \
      2   3        ← 第2层
     / \   \
    4   5   6      ← 第3层

BFS 遍历结果：1, 2, 3, 4, 5, 6

BFS 的队列变化：

步骤 队列状态 访问节点
初始 [1] -
1 [2, 3] 1
2 [3, 4, 5] 2
3 [4, 5, 6] 3
4 [5, 6] 4
5 [6] 5
6 [] 6

适用场景：

  • 寻找最短路径（无权图）
  • 按层次打印树结构
  • 查找距离根节点最近的满足条件的节点

DFS（深度优先搜索）

核心思想：

  • 沿着一条分支一直走到底，然后回溯
  • 使用栈（Stack）或递归实现

三种遍历方式的区别在于访问根节点的时机：

1. 前序遍历（Pre-order）

访问顺序：根节点 → 左子树 → 右子树

口诀：根左右

遍历过程（以上述示例树）：

  1. 访问根 1
  2. 遍历左子树（以 2 为根）
    • 访问 2
    • 遍历 2 的左子树（访问 4）
    • 遍历 2 的右子树（访问 5）
  3. 遍历右子树（以 3 为根）
    • 访问 3
    • 遍历 3 的左子树（空）
    • 遍历 3 的右子树（访问 6）

前序遍历结果：1, 2, 4, 5, 3, 6

递归实现：

void preorder(TreeNode root) {
    if (root == null) return;
    System.out.print(root.val + " ");  // 访问根
    preorder(root.left);               // 遍历左子树
    preorder(root.right);              // 遍历右子树
}

适用场景：

  • 复制一棵树（先创建根节点）
  • 序列化/反序列化树结构
  • 计算目录大小（先算当前目录，再算子目录）
2. 中序遍历（In-order）

访问顺序：左子树 → 根节点 → 右子树

口诀：左根右

遍历过程：

  1. 遍历左子树（以 2 为根）
    • 遍历 2 的左子树（访问 4）
    • 访问 2
    • 遍历 2 的右子树（访问 5）
  2. 访问根 1
  3. 遍历右子树（以 3 为根）
    • 遍历 3 的左子树（空）
    • 访问 3
    • 遍历 3 的右子树（访问 6）

中序遍历结果：4, 2, 5, 1, 3, 6

递归实现：

void inorder(TreeNode root) {
    if (root == null) return;
    inorder(root.left);                // 遍历左子树
    System.out.print(root.val + " ");  // 访问根
    inorder(root.right);               // 遍历右子树
}

适用场景：

  • 二叉搜索树（BST）的中序遍历结果是有序的
  • 计算表达式树（中缀表达式）
  • 从小到大输出 BST 的所有节点

重要性质：

对于二叉搜索树，中序遍历得到升序序列。这是 BST 的核心性质，常用于验证一棵树是否为 BST。

3. 后序遍历（Post-order）

访问顺序：左子树 → 右子树 → 根节点

口诀：左右根

遍历过程：

  1. 遍历左子树（以 2 为根）
    • 遍历 2 的左子树（访问 4）
    • 遍历 2 的右子树（访问 5）
    • 访问 2
  2. 遍历右子树（以 3 为根）
    • 遍历 3 的左子树（空）
    • 遍历 3 的右子树（访问 6）
    • 访问 3
  3. 访问根 1

后序遍历结果：4, 5, 2, 6, 3, 1

void postorder(TreeNode root) {
    if (root == null) return;
    postorder(root.left);              // 遍历左子树
    postorder(root.right);             // 遍历右子树

  • 逆波兰表达式（后缀表达式）求值

四种遍历对比总结

遍历方式 访问顺序 遍历结果 核心应用
BFS 层次遍历 1, 2, 3, 4, 5, 6 最短路径、层次打印
先序 根左右 1, 2, 4, 5, 3, 6 树复制、序列化
中序 左根右 4, 2, 5, 1, 3, 6 BST 排序、中缀表达式
后序 左右根 4, 5, 2, 6, 3, 1 删除树、后缀表达式

记忆技巧

DFS 三种遍历的统一记忆法：

所有 DFS 遍历都按左子树 → 右子树的顺序，区别只在根节点的位置：

遍历 根的位置 记忆
先序 最前面（第一位） 根在最"先"
中序 中间位置 根在"中"间
后序 最后面（末位） 根在最"后"

快速判断遍历结果的方法：

  1. 找根节点：

    • 先序：第一个元素是根
    • 后序：最后一个元素是根
    • 中序：根在中间，左边是左子树，右边是右子树

  2. 递归分解：

    • 确定根后，在中序中找到根的位置
    • 左边是左子树的中序，右边是右子树的中序
    • 根据子树长度，在先序/后序中划分对应部分
    • 递归处理子树

// 根据前序和中序求后序
void buildPostorder(int[] preorder, int preStart, int preEnd,
                    int[] inorder, int inStart, int inEnd,
                    List<Integer> postorder) {
    if (preStart > preEnd || inStart > inEnd) return;
    
    // 前序第一个元素是根
    int rootVal = preorder[preStart];
    
    // 在中序中找到根的位置
    int rootIndex = -1;
    for (int i = inStart; i <= inEnd; i++) {
        if (inorder[i] == rootVal) {
            rootIndex = i;
            break;
        }
    }
    
    // 左子树节点数
    int leftSize = rootIndex - inStart;
    
    // 递归构建左子树的后序（前序跳过根，取 leftSize 个；中序取左边 leftSize 个）
    buildPostorder(preorder, preStart + 1, preStart + leftSize,
                   inorder, inStart, rootIndex - 1,
                   postorder);
    
    // 递归构建右子树的后序（前序取后面；中序取右边）
    buildPostorder(preorder, preStart + leftSize + 1, preEnd,
                   inorder, rootIndex + 1, inEnd,
                   postorder);
    
    // 添加根节点（后序：左右根，根最后）
    postorder.add(rootVal);
}

关键规律总结：

遍历类型 根的位置 作用
前序 第一个元素 用于确定根
中序 根在中间，左右分割 用于确定左右子树范围
后序 最后一个元素 根最后，递归拼接结果

核心技巧：前序找根，中序分左右，递归求子树后序，最后拼上根。

boolean inorderCheck(TreeNode node, long min, long max) {
    if (node == null) return true;
    if (node.val <= min || node.val >= max) return false;
    return inorderCheck(node.left, min, node.val) 
        && inorderCheck(node.right, node.val, max);
}

软件生命周期

什么是软件生命周期

软件生命周期（Software Development Life Cycle, SDLC）是指软件从概念提出到最终退役的全过程，包括规划、开发、运行和维护的各个阶段。

经典瀑布模型（Waterfall Model）

瀑布模型是最早的软件开发模型，将生命周期划分为线性顺序的阶段：

阶段 主要任务 输出物
需求分析 收集用户需求，确定系统功能和非功能需求 需求规格说明书（SRS）
系统设计 概要设计（架构）、详细设计（模块接口、算法） 设计文档、UML 图
编码实现 根据设计文档编写代码 源代码、单元测试
测试验证 集成测试、系统测试、验收测试 测试报告、Bug 列表
部署运维 软件发布、安装配置、运行维护 部署文档、运维手册
退役 系统下线、数据迁移、资源回收 退役报告

各阶段耗时与成本占比

根据软件工程统计数据（行业平均值）：

阶段 耗时占比 成本占比 说明

后期修复缺陷的成本是前期的 10-100 倍

因果图与判定表

什么是因果图

因果图（Cause-Effect Graph） 是一种用于黑盒测试的用例设计方法，通过分析输入条件（原因）和输出结果（结果）之间的逻辑关系，系统地生成测试用例。

核心思想：

  • 将程序的输入条件视为因（Cause）
  • 将程序的输出动作或结果视为果（Effect）
  • 分析因与果之间的逻辑关系（与、或、非、恒等、约束等）
  • 将因果图转换为判定表，进而生成测试用例

适用场景：

  • 多输入条件组合复杂的情况
  • 需要检查条件之间的约束关系
  • 业务规则复杂、条件之间存在依赖的系统

因果图的图形符号

符号 名称 含义 图示
Ci 原因节点 输入条件，取值为 0（假）或 1（真） 圆形
Ei 结果节点 输出结果，取值为 0 或 1 圆形
∧ 与（AND） 所有原因都为真时，结果为真 原因 → ∧ → 结果
∨ 或（OR） 任一原因为真时，结果为真 原因 → ∨ → 结果
¬ 非（NOT） 原因取反 原因 → ¬ → 结果
= 恒等 原因直接决定结果 原因 = 结果

因果图的约束关系

约束用于描述原因之间或结果之间的限制条件：

1. 原因之间的约束（输入约束）：

约束 符号 含义 示例
E（互斥） E 至多一个为真，可以同时为假 单选按钮
I（包含） I 至少一个为真，可以同时为真 多选必填
O（唯一） O 必须且只能一个为真 单选必填
R（要求） R 一个为真，另一个也必须为真 联动条件
M（强制） M 一个为真，另一个必须为假 互斥条件

2. 结果之间的约束（输出约束）：

约束 符号 含义
R（要求） R 一个结果为真，另一个也必须为真
M（强制） M 一个结果为真，另一个必须为假

什么是判定表

判定表（Decision Table） 是将因果图转换为表格形式，列出所有可能的条件组合及对应的结果，是生成测试用例的直接依据。

判定表的结构：

桩 规则 1 规则 2 规则 3 … 规则 n
条件桩
条件 1 T T F … F
条件 2 T F T … F
… … … … … …
动作桩
动作 1 √ √ × … ×
动作 2 × √ √ … ×
… … … … … …
  • 条件桩：列出所有输入条件
  • 动作桩：列出所有输出结果
  • 规则列：每一列代表一组条件组合及对应动作

因果图与判定表的转换步骤

步骤 1：识别原因和结果

分析需求规格说明书，提取：

  • 所有输入条件（原因）
  • 所有输出结果（结果）

步骤 2：分析因果关系

  • 哪些原因组合会产生哪些结果
  • 原因之间是否存在约束

步骤 3：画出因果图

使用图形符号表示因果关系和约束。

步骤 4：转换为判定表

将因果图转换为判定表，每一列对应一个测试用例。

步骤 5：简化判定表

  • 合并相似的规则（使用"-“表示无关条件）
  • 去除不可能的组合（考虑约束条件）

步骤 6：生成测试用例

每一列对应一个测试用例。

因果图与判定表的优缺点

优点：

优点 说明
系统化 系统地覆盖所有条件组合，避免遗漏
发现隐藏缺陷 能够发现条件组合触发的隐藏 Bug
减少冗余 通过约束简化，减少无效测试用例
文档化 清晰的测试依据，便于维护

缺点：

缺点 说明
组合爆炸 条件多时（>5个），组合数呈指数增长（2^n）
耗时 分析复杂业务规则需要较多时间
不适合所有场景 输入输出关系简单时，性价比不高

笔试常见考点

考点 1：因果图的约束类型

问：因果图中 E 约束和 O 约束的区别？

答：

  • E 约束（互斥）：至多一个为真，可以同时为假
  • O 约束（唯一）：必须且只能一个为真，不能同时为假

考点 2：判定表的组成

问：判定表由哪几部分组成？

答：条件桩、动作桩、规则列。条件桩列出所有输入条件，动作桩列出所有输出动作，规则列表示条件组合与对应动作。

考点 3：简化判定表的方法

问：如何简化判定表？

答：

  1. 使用”-“表示无关条件，合并相似规则
  2. 根据约束关系去除不可能的组合
  3. 合并产生相同动作的规则

考点 4：因果图 vs 等价类划分 vs 边界值分析

方法 适用场景 关注点
等价类划分 输入条件独立 减少用例数量，每个等价类选一个代表
边界值分析 输入范围明确 边界上的错误概率高
因果图 条件之间有依赖 条件组合的逻辑关系
判定表 条件组合复杂 系统化列出所有组合
