Project

deep2X 后端解析

deep2X 是一套基于 Spring Boot 3 + Spring Cloud + Spring AI 的企业级 RAG 知识库问答微服务平台，本文从项目架构、模块职责、RAG 核心流程、对话系统到基础设施配置进行全面解析。

2026-04-05

项目概述

deep2X（代号 knowledge）是一套企业级 AI 知识库问答平台后端服务。用户可以将企业文档（PDF、Word、文本等）上传并切分成文本块（Chunk），系统自动将其向量化存入 PostgreSQL/pgvector，供后续的语义检索使用；前台用户通过与"智能体（Robot）“对话，系统借助 RAG（检索增强生成）从知识库中召回相关内容，结合 LLM 生成准确、有依据的回答。

核心价值：

  • 企业文档知识的沉淀与复用：上传一次，多次问答
  • 多模型支持：同时接入 DeepSeek、OpenAI、通义千问等大模型
  • 三种检索策略：向量检索、全文检索、混合检索，灵活配置召回精度
  • 智能体配置化：每个 Robot 可独立配置关联知识库、召回策略、Prompt、模型等
  • 微服务架构：各功能模块独立部署，通过 Nacos 注册发现，Gateway 统一鉴权

整体架构

模块拓扑

架构分为四层，由上至下依次为：

接入层

  • cloud-gateway：所有外部请求的统一入口，负责 sa-token 鉴权、CORS 跨域处理、全局请求日志，通过 Nacos 服务发现将请求路由至下游服务。

业务层（通过 Nacos 注册发现，由网关路由）

  • cloud-system：系统管理服务，提供用户、部门、角色、菜单、OpenAPI 密钥等基础能力，被其他业务服务通过 OpenFeign 调用。
  • cloud-business/knowledge-agent：后台管理核心，负责知识库管理、智能体（Robot）配置、AI 模型/厂商管理、文本向量化任务调度；通过 Feign 调用 cloud-system 查询用户/部门信息。
  • cloud-business/agent-rest：前台接口层，面向终端用户，提供知识库检索（三种模式）、流式对话（SSE）、智能体 RAG 对话、联网搜索等接口。
  • cloud-business/consumer：消息消费服务，监听 RabbitMQ 队列，执行耗时的文本 Embedding 向量化任务，将结果写回 PostgreSQL。

存储层

  • MySQL：存储系统管理数据（sys_user、sys_role、sys_menu 等）。
  • PostgreSQL + pgvector：存储知识库业务数据和向量数据（cw_text_vector_task、cw_kb_robot、cw_ai_manufacturer 等），embedding VECTOR 列用于相似度检索。
  • Redis：存储 sa-token 登录 Token（分布式会话共享）和热点数据缓存。

消息队列

  • RabbitMQ：解耦向量化任务，knowledge.exchange 路由至 knowledge.queue（主队列），消费失败超过 3 次后转入 knowledge.dead.letter.queue（死信队列）记录告警。

// VectorUtil.java - 核心方法
public static List<Double> normalizeVectorWithND4J(List<Double> vector) {
    double[] array = vector.stream().mapToDouble(Double::doubleValue).toArray();
    INDArray ndArray = Nd4j.create(array);
    double norm = ndArray.norm2Number().doubleValue();
    
    // 处理零向量：避免除以零
    if (Math.abs(norm) < 1e-12) {
        return vector.stream().map(d -> 0.0).collect(Collectors.toList());
    }
    
    INDArray normalized = ndArray.div(norm);
    return Arrays.stream(normalized.toDoubleVector()).boxed().collect(Collectors.toList());
}

为什么归一化：在做余弦相似度检索时，归一化后的向量点积就等于余弦相似度，pgvector 的 <#> 操作符（内积）在归一化向量上等价于余弦相似度，提升检索精度与一致性。

RAG 核心流程（重点）

为什么用 RAG，不直接问 LLM？

问题 直接问 LLM RAG
知识时效性 训练数据截止日，不知道最新文档 实时读取知识库，始终是最新内容
幻觉（虚假信息） 高，可能编造没有的事实 低，回答有明确的文档支撑
私有知识 无法使用企业内部文档 完全支持自定义知识库
上下文长度 受 Token 窗口限制，无法传入全量文档 将相关内容检索后再传入，显著节省 Token

// MessageServiceImpl.java - processEmbeddingMessage()
private void processEmbeddingMessage(MessageDto messageDto) {
    // 1. 根据任务ID查询任务详情及模型配置（联表查询）
    TextVectorTaskVo task = vectorTaskMapper.taskInfoById(messageDto.getContent());

    List<Double> embedding = new ArrayList<>();
    boolean isSuccess = true;

    // 2. 根据厂商代码选择对应的 Embedding API
    switch (task.getManufacturerCode()) {
        case "openai":
            // POST {baseUrl}/v1/embeddings
            embedding = vectorService.openaiEmbedding(
                task.getContent(), task.getModelName(),
                task.getKey(), task.getManufacturerUrl());
            break;
        case "qwen":
            // POST {baseUrl}/compatible-mode/v1/embeddings，dimension=1024
            embedding = vectorService.qwenEmbedding(
                task.getContent(), task.getModelName(),
                task.getKey(), task.getManufacturerUrl(), 0);
            break;
        default:
            isSuccess = false;
    }

    // 3. L2 归一化，增强向量相似度计算精度
    embedding = VectorUtil.normalizeVectorWithND4J(embedding);

    // 4. 写回数据库
    TextVectorTask saveTask = new TextVectorTask();
    saveTask.setId(messageDto.getContent());
    if (isSuccess) {
        // 格式：'[0.1, -0.3, ...]'::vector
        String embeddingStr = embedding.stream()
            .map(String::valueOf)
            .collect(Collectors.joining(",", "[", "]"));
        saveTask.setEmbedding(embeddingStr);
        saveTask.setState("1");  // 成功
    } else {
        saveTask.setState("2");  // 失败
    }
    vectorTaskMapper.updateEmbeddingById(saveTask);
}

@PostMapping(value = "/sseAgentChat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ChatResponse> sseAgentChat(@RequestBody AgentChatDto agentDto) {
    LoginUser loginUser = LoginUserUtil.getUserInfo();

    // 1. 获取会话历史记录
    ChatDto chatDto = agentSessionService.getSessionChatRecord(agentDto.getSessionId());

    // 2. 获取智能体对应的 LLM 模型实例
    StreamingChatModel deepChatModel =
        agentService.getOpenAiChatModelByRobotId(agentDto.getAgentId());

    // 3. 构建 Prompt 消息列表
    List<Message> messages = new ArrayList<>();

    // 3.1 注入历史对话（多轮会话能力）
    for (KbRobotRecord record : chatDto.getChatRecords()) {
        messages.add(new UserMessage(record.getAsk()));
        if (StringUtils.isNotBlank(record.getReply())) {
            messages.add(new AssistantMessage(record.getReply()));
        }
    }

    // 3.2 注入知识库检索结果（RAG 核心）
    if (!CollectionUtils.isEmpty(agentDto.getVectorInfos())) {
        StringBuilder knowInfos = new StringBuilder(
            "以下信息是跟用户本次提问相关的知识请，对其加以分析回答用户问题：");
        for (TextVectorTaskVo item : agentDto.getVectorInfos()) {
            if (StringUtils.isEmpty(item.getAnswer())) {
                // 普通文本 chunk
                knowInfos.append(StringUtils.join(item.getContent(), ";"));
            } else {
                // QA 类型：直接给出问题-答案对
                knowInfos.append("问题：").append(item.getContent())
                         .append("，答案：").append(item.getAnswer());
            }
        }
        messages.add(new SystemMessage(knowInfos.toString()));
    }

    // 3.3 添加当前用户消息
    messages.add(new UserMessage(agentDto.getMessage()));

    // 4. 流式调用 LLM，实时返回 SSE 事件流
    StringBuilder fullResponse = new StringBuilder();
    return deepChatModel.stream(new Prompt(messages))
        .doOnNext(response -> {
            fullResponse.append(response.getResult().getOutput().getText());
        })
        .doOnComplete(() -> {
            // 5. 流式结束后异步保存对话记录
            agentSessionService.saveChatRecordToSession(
                agentDto.getSessionId(), agentDto.getAgentId(),
                agentDto.getMessage(), fullResponse.toString(),
                agentDto.getVectorInfos());
        });
}

三种检索模式对比

检索模式 原理 适用场景 pgvector 操作符
向量检索（VECTOR） 将问题向量化，与存储向量做相似度计算 语义理解、换句话问同一个问题 <#> 内积（归一化后等价余弦）
全文检索（FULL_TEXT） HanLP 中文分词 → to_tsquery → PostgreSQL FTS 精确词汇匹配、关键字检索 @@ 全文匹配 + ts_rank 排序
混合检索（HYBRID） 向量得分 × vectorWeight + 文本得分 × textWeight × 10 平衡语义理解和精确匹配 组合使用，支持相似度阈值过滤

-- 向量检索（余弦相似度，1-内积距离，越大越相似）
SELECT *, 1 - (embedding <#> :vector) AS similarity
FROM cw_text_vector_task
WHERE know_id = :knowId AND deleted = 0 AND state = '1'
  AND 1 - (embedding <#> :vector) >= :similarity
ORDER BY similarity DESC
LIMIT :limit;

-- 全文检索（HanLP 分词构建 ts_query，短文本用 & 精确匹配，长文本用 | 宿松匹配）
SELECT *, ts_rank(to_tsvector('chinese_zh', content),
                 to_tsquery('chinese_zh', '产品 & 价格')) AS rank
FROM cw_text_vector_task
WHERE know_id = :knowId AND deleted = 0
  AND to_tsvector('chinese_zh', content) @@
      to_tsquery('chinese_zh', :question)
ORDER BY rank DESC LIMIT :limit;

-- 混合检索（向量语义得分 + 全文关键字得分加权融合）
SELECT *, 
  :vectorWeight * (1 - (embedding <=> :vector)) +
  :textWeight * 10 * ts_rank(to_tsvector('chinese_zh', content),
                              to_tsquery('chinese_zh', :question)) AS score
FROM cw_text_vector_task
WHERE know_id = :knowId AND deleted = 0 AND state = '1'
  AND (1 - (embedding <=> :vector)) >= :similarity
ORDER BY score DESC LIMIT :limit;

pgvector 操作符说明：

操作符 含义 返回值 本项目使用
<#> 内积距离 内积的负值（越小越相似） 向量检索：1 - (embedding <#> :vector) 转换为相似度
<=> 余弦距离 0~2（越小越相似） 混合检索：1 - (embedding <=> :vector) 转为相似度
<-> 欧氏距离 绝对距离 未使用（不适合归一化向量）

经过 L2 归一化的向量，内积等价于余弦相似度。<#> 在归一化向量上速度更快（无需除以模长）。

混合检索中 ×10 的作用：ts_rank 返回的全文得分通常在 0.001 到 0.1 范围内，向量相似度在 0 到 1 范围内。直接相加会让全文分几乎没有贡献，乘以 10 进行量纲对齐，使两种得分处于可比较的数量级，让加权确实起作用。

@RabbitListener(queues = RabbitMQConfig.QUEUE_NAME)
public void handleMessage(MessageDto messageDto, Channel channel,
                          @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
    try {
        messageService.processMessage(messageDto);
        channel.basicAck(tag, false);  // ✅ 处理成功，手动确认
    } catch (Exception e) {
        int retryCount = messageDto.getRetryCount() == null ? 0 : messageDto.getRetryCount();
        if (retryCount < 3) {
            // 未超限：重新投递，原消息手动 ACK
            messageDto.setRetryCount(retryCount + 1);
            rabbitTemplate.convertAndSend(EXCHANGE_NAME, ROUTING_KEY, messageDto);
            channel.basicAck(tag, false);
        } else {
            // 超过 3 次：原消息 NACK 不重回主队列 → 自动转入死信队列
            channel.basicNack(tag, false, false);
        }
    }
}

若使用 acknowledge-mode: auto（默认），消费者方法抛出异常时消息会立即重回队列并反复重试，在 Embedding API 超时等场景下会造成消息积压风暴。必须配置手动 ACK 或 Spring Retry 的退避策略。

            // reasoningContent：思考过程（<think>...</think>）
            if (StringUtils.isNotBlank(msg.getReasoningContent())) {
                reason.append(msg.getReasoningContent());
            }
            // text：最终回答
            if (StringUtils.isNotBlank(msg.getText())) {
                fullResponse.append(msg.getText());
            }
        });
}

保存时 reason（思维链）和 fullResponse（最终回答）分别存储在 cw_chat_record.reasoning 和 cw_chat_record.reply。

踩坑与难点（用户问答流程）

以下三个坑均出现在 POST /agent/sseAgentChat 的完整请求链路中，考察的是 Java 并发、Spring 事务原理和响应式编程生命周期。

客户端中途断开导致对话记录静默丢失

定位问题：测试 SSE 对话接口时，用户反馈"AI 已经回答了一半，不小心关了页面，再打开发现历史记录里没有这条对话”。查数据库确认记录缺失，查服务端日志没有任何报错。在 doOnComplete() 中加打印日志，发现关闭页面场景下这行日志根本没有输出，说明 doOnComplete() 根本没有被执行到。

解决尝试：排查前端关闭逻辑后确认前端正常，问题在服务端。查阅 Reactor 文档得知 Flux 有三种终止状态：onComplete（正常结束）、onError（异常）、onCancel（订阅取消）。客户端断开 SSE 连接会触发 onCancel，而 doOnComplete() 只监听 onComplete，对 cancel 无感知。

尝试在 doOnError() 中也加保存逻辑——无效，cancel 既不走 onError 也不走 onComplete，两个钩子都不会触发。

最终效果：改用 doFinally(signalType -> {...})，该钩子在所有三种终止状态下均触发，SignalType 参数可区分是正常结束、报错还是取消：

1

3
4
5

7

return chatModel.stream(prompt)
    .doOnNext(chunk -> fullResponse.append(chunk.getResult().getOutput().getText()))
    .doFinally(signalType -> {
        // ON_COMPLETE / ON_ERROR / CANCEL 均会触发
        if (fullResponse.length() > 0) {
            saveRecord(fullResponse.toString());  // ✅ 有内容就保存，不丢记录
        }
    });

上线后无论用户正常结束、网络中断还是主动关闭页面，只要 LLM 有输出就会保存，记录丢失问题彻底解决。

@Transactional 在 Reactor 异步线程中失效

定位问题：解决记录丢失后，发现保存时偶发两张表数据不一致：cw_chat_record 表有记录，但 cw_agent_session 的最后对话时间没有同步更新，或反过来。手动制造保存失败场景，确认写 A 表成功后写 B 表失败时，A 表的数据没有回滚，说明两步写库操作没有在同一事务中。

解决尝试：确认 saveRecord() 有 @Transactional 注解，且通过注入的 Spring Bean 调用，不存在同类调用问题。打印执行线程名，发现 doFinally() 在 reactor-http-epoll-x 线程上执行，不是 Spring MVC 的原始请求线程。查阅 Spring 源码确认：@Transactional 底层用 TransactionSynchronizationManager 将数据库 Connection 存入 ThreadLocal，Reactor 切换线程后新线程上没有这个上下文，声明式事务的提交/回滚行为不可控。

尝试用 .subscribeOn(Schedulers.single()) 将 doFinally 切回固定线程——Reactor 调度时机难以保证，行为不稳定，仍有概率失效。

最终效果：在 doFinally() 内改用 TransactionTemplate 编程式事务，在当前线程上主动建立事务上下文，不依赖 ThreadLocal 传播：

 1
 2
 3
 4
 5

 7

transactionTemplate.execute(status -> {
    try {
        recordMapper.insert(...);   // 写 A 表
        sessionMapper.update(...);  // 写 B 表
    } catch (Exception e) {
        status.setRollbackOnly();   // 手动标记回滚
        throw e;
    }
    return null;
});

改造后两张表始终保持一致，任意一步失败都能正确回滚，数据一致性问题彻底解决。

StringBuilder 拼接流式响应的线程安全问题

定位问题：对接口进行 500 并发压测时，数据库中少量对话记录的回答内容出现截断或乱码拼接，概率约 0.3%，单次手动测试无法复现。增大压测并发数后概率上升，判断是并发问题。打印每次保存前的 fullResponse 内容，确认入库前数据就已经错误，排除数据库侧问题。

解决尝试：排查 LLM API 返回日志，确认模型输出完整，排除上游问题。审查 doOnNext() 中对 fullResponse.append() 的调用，注意到为提升吞吐量引入了 .publishOn(Schedulers.boundedElastic()) 后，doOnNext 可能在不同线程上被调用。StringBuilder.append() 内部没有同步，并发扩容时 char[] 的复制和写入之间会发生线程切换，导致数据被覆盖或字段丢失。

尝试用 synchronized (fullResponse) { fullResponse.append(...); } 手动加锁——有效，但每处 append() 都要包裹，写法繁琐且容易遗漏。

最终效果：将 StringBuilder 替换为 StringBuffer，其 append() 方法内置 synchronized，语义等价于手动加锁的 StringBuilder，修改一行代码解决问题：

1
2
3
4

// 修改前
StringBuilder fullResponse = new StringBuilder();
// 修改后
StringBuffer fullResponse = new StringBuffer();  // append() 内置 synchronized

压测 10000 次后记录内容全部正确，问题不再复现。同时在团队中建立规范：Reactor 异步流回调 lambda 中对被捕获的共享可变变量，必须使用线程安全类型。
