Redis 系统学习笔记

从基础数据类型、底层数据结构到持久化、高可用、缓存设计模式的 Redis 全栈学习笔记，结合 hmdp 点评项目的真实应用场景进行讲解。

2026-04-05

概述

Redis（Remote Dictionary Server）是一款基于内存的高性能键值数据库，支持多种数据结构，具备持久化、主从复制、哨兵、集群等企业级特性。

核心特点：

  • 纯内存操作，读写速度可达 10 万+ QPS
  • 单线程命令处理（6.0 后 I/O 多线程），无锁竞争
  • 丰富的数据结构：String、Hash、List、Set、ZSet 及 GEO、BitMap、Stream 等扩展类型
  • 支持 Lua 脚本原子执行、事务、发布订阅
  • 持久化（RDB + AOF）+ 集群保障高可用

ZSet（有序集合）

ZSet 是 Redis 最强大的数据类型，每个成员绑定一个浮点数 score，按 score 升序自动排列，天然支持排行榜、范围查询、游标分页。

内部编码

编码 条件
listpack（Redis 7.0+）/ ziplist（旧版） 元素数 ≤ zset-max-listpack-entries（默认 128）且每个成员 ≤ 64 字节
skiplist + hashtable 超出阈值（跳表保证范围查询 O(log N)，hashtable 保证按成员查 score O(1)）

// BITFIELD GET 取本月所有签到位，以 dayOfMonth 位无符号整数读取
List<Long> result = stringRedisTemplate.opsForValue().bitField(
    "sign:" + userId + ":" + yearMonth,
    BitFieldSubCommands.create()
        .get(BitFieldSubCommands.BitFieldType.unsigned(day)).valueAt(0)
);
long bits = result.get(0);
int streak = 0;
while ((bits & 1) != 0) {  // 从最低位（今天）向高位（早期日期）统计
    streak++;
    bits >>>= 1;
}

BITFIELD 的 unsigned(dayOfMonth) 读取的是高 dayOfMonth 位，最低位对应今天。右移配合 & 1 可以高效逐位判断，无需 GETBIT 逐位查询。

底层数据结构

SDS（Simple Dynamic String，简单动态字符串）

Redis 的字符串不直接使用 C 原生 char*，而是自定义 SDS 结构：

1
2
3
4
5
6

struct sdshdr {
    int len;     // 已使用字节数，O(1) 获取长度
    int alloc;   // 已分配总字节数（不含头部和结尾 \0）
    char flags;  // 编码类型（sdshdr8/16/32/64）
    char buf[];  // 实际字节数组
};

SDS 相对 C 字符串的优势：

  • O(1) 获取字符串长度（直接读 len 字段，无需遍历）
  • 预分配：修改时若空间不足，分配额外空间（len < 1MB 则翻倍，≥ 1MB 则多分配 1MB），减少扩容次数
  • 惰性释放：缩短字符串时不立即释放多余空间，减少重新分配
  • 二进制安全：通过 len 判断结束，而非 \0，可存任意二进制数据

SkipList（跳表）

ZSet 的范围查询由跳表实现，跳表是在有序链表基础上加了多级索引，实现 O(log N) 的查找和插入：

1
2
3
4

Level 4: head ──────────────────────── 50 ──── tail
Level 3: head ────── 20 ──────────────── 50 ──── tail
Level 2: head ── 10 ── 20 ────── 35 ──── 50 ──── tail
Level 1: head ── 10 ── 20 ── 28 ── 35 ── 50 ── 60 ── tail
  • 查找：从最高层索引快速跳跃，逐层缩小范围，最终在底层找到目标 O(log N)
  • 插入：随机决定节点层数（越高层概率越低，期望层数 O(log N)），更新各层指针
  • 相比平衡树：实现更简单，范围查询性能相当，且跳表的修改不需要复杂的旋转操作

Redis ZSet 同时维护一个 HashTable（成员 → score 的映射），使 ZSCORE 达到 O(1)；跳表负责 ZRANGE、ZRANK 等范围和排名查询，复杂度为 O(log N)；ZRANGE 返回 M 个结果时总复杂度为 O(log N + M)。

HashTable（哈希表）

Redis HashTable 是一个数组 + 链表（拉链法）解决冲突的结构，同时维护两张哈希表用于渐进式 rehash：

1

3
4

    dictht ht[2];   // 两张哈希表，平时只用 ht[0]
    int rehashidx;  // -1 = 未在 rehash；≥0 = 正在 rehash 的桶下标
}

渐进式 rehash（解决 rehash 阻塞问题）：

  • 触发条件：负载因子（used / size）> 1 时扩容（有 BGSAVE 时阈值提升到 5），< 0.1 时缩容
  • 扩容时不一次性迁移所有数据，而是每次增删改查操作时迁移 rehashidx 指向的一个桶
  • rehash 期间读写同时查两张表，rehash 完成后释放旧表

键过期与内存淘汰

过期键删除策略

Redis 使用惰性删除 + 定期删除的组合策略：

策略 原理 优点 缺点
惰性删除 访问 Key 时才检查是否过期 CPU 友好，只在使用时检查 过期 Key 长期不被访问导致内存泄漏
定期删除 每隔一定时间随机抽取部分 Key 检查删除 兼顾内存和 CPU 无法保证及时清理所有过期 Key

Redis 每 100ms 执行一次定期扫描，随机抽取 20 个 Key，若过期比例 > 25% 则继续抽取，直到比例降低或超过 25ms 时间限制。

内存淘汰策略（maxmemory-policy）

当内存达到 maxmemory 上限时，根据策略淘汰数据：

策略 描述 适用场景
noeviction（默认） 拒绝写入，返回错误 不允许丢数据时
allkeys-lru 淘汰所有 Key 中最近最少使用的 通用缓存场景
volatile-lru 只淘汰设了过期时间的 Key（LRU） 缓存与持久数据混存
allkeys-random 随机淘汰所有 Key 均匀访问时
volatile-random 随机淘汰有过期时间的 Key -
volatile-ttl 淘汰剩余 TTL 最短的 Key 按即将过期优先清理
allkeys-lfu 淘汰所有 Key 中访问频率最低的（Redis 4.0+） 热点数据差异明显时
volatile-lfu 只淘汰设了过期时间的 Key（LFU） -

纯缓存场景推荐 allkeys-lru 或 allkeys-lfu；volatile-lru 适合缓存和持久数据混合存储，避免淘汰不应删除的数据。

持久化

RDB（快照）

RDB 将某一时刻的全量数据序列化为二进制文件（dump.rdb），支持手动触发和自动触发：

1
2

SAVE    # 同步，阻塞主线程（慎用）
BGSAVE  # 异步，fork 子进程执行，主线程不阻塞

自动触发配置（redis.conf）：

1
2
3

save 3600 1     # 1 小时内至少 1 次写操作
save 300 100    # 5 分钟内至少 100 次写操作
save 60 10000   # 1 分钟内至少 10000 次写操作

BGSAVE + COW（写时复制）：fork 子进程与父进程共享内存页，写时才复制修改的页，保证快照数据的一致性，同时主进程可继续处理请求。

优点：文件紧凑，恢复速度快，适合备份。

缺点：两次快照之间的数据可能丢失（丢失窗口 = save 间隔）。

-- seckill.lua（hmdp 秒杀资格原子判断）
local voucherId = ARGV[1]
local userId    = ARGV[2]
local stockKey  = 'seckill:stock:' .. voucherId
local orderKey  = 'seckill:order:' .. voucherId

-- 1. 库存不足
if tonumber(redis.call('get', stockKey)) <= 0 then return 1 end
-- 2. 重复下单
if redis.call('sismember', orderKey, userId) == 1 then return 2 end
-- 3. 原子扣减库存 + 记录下单
redis.call('decr', stockKey)
redis.call('sadd', orderKey, userId)
return 0

事务

Redis 事务通过 MULTI / EXEC 将命令打包执行，但不支持回滚（执行时命令报错不会中断其余命令）：

1

MULTI       # 开启事务
SET k1 v1
INCR k2
EXEC        # 提交，按顺序执行所有命令
DISCARD     # 放弃事务
WATCH key   # 乐观锁：若 EXEC 前 key 被其他客户端修改，事务放弃执行

Redis 事务没有原子性保证：若某条命令执行报错（如对 String 执行 INCR），其他命令仍正常执行，不会回滚。真正需要原子性请使用 Lua 脚本。

性能优化

避免大 Key

大 Key（String > 10KB，Hash/List/Set/ZSet 元素 > 5000 个）会导致：

  • 阻塞主线程：序列化/反序列化耗时，KEYS *、SMEMBERS 等全量查询卡住
  • 内存碎片：大对象分配不均匀
  • 迁移慢：Cluster 模式下大 Key 所在 slot 迁移耗时极长

处理方式：大 String 拆分存储；大 Hash 按字段 hash 分片（如 user:1:profile:0 到 user:1:profile:n）；大集合用 HSCAN/SSCAN/ZSCAN 分批遍历。

避免热点 Key

单一 Key 请求量过大（如秒杀商品库存、热门博文点赞）会打满单节点带宽。解决方案：

  • 本地缓存：Caffeine 等本地缓存缓冲热点请求，减少 Redis 压力
  • Key 分片：将热点 Key 拆为 hot:key:0 到 hot:key:N，随机读取，写时聚合
  • 读写分离：热点读走从节点
