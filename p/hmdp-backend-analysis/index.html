hmdp 后端解析

hmdp 是一个基于 Spring Boot 2 + Redis 的点评系统，本文从项目概述、用户鉴权、缓存设计、秒杀系统到社交功能进行全面解析，覆盖所有核心技术细节与实现代码。

2026-04-05

项目概述

hmdp（hm-dianping）是一个仿大众点评的单体 Spring Boot 应用，用户可以浏览商铺、发布图文点评、为点评点赞、领取优惠券并参与秒杀抢购，同时支持关注博主订阅其动态。项目以 Redis 为核心，系统性地实践了缓存设计、分布式锁、全局唯一 ID、消息队列等主流 Redis 应用场景。

核心技术亮点：

  • 手机验证码 + Redis Token 登录，双拦截器链实现请求鉴权
  • Redis 缓存三大问题（穿透、击穿、雪崩）的多种解决方案并存（注释对比）
  • Lua 脚本原子化秒杀资格判断，RabbitMQ 消息队列异步落库
  • Redisson 分布式锁保证一人一单
  • Redis ZSet 实现点赞排行榜、Feed 流推送、附近商铺（GEO）、用户签到（BitMap）
  • 基于 Redis 自增的全局唯一 ID 生成器（时间戳 + 序列号位拼接）

// UserHolder.java
public class UserHolder {
    private static final ThreadLocal<UserDTO> tl = new ThreadLocal<>();
    public static void saveUser(UserDTO user){ tl.set(user); }
    public static UserDTO getUser(){ return tl.get(); }
    public static void removeUser(){ tl.remove(); }  // afterCompletion 中清理，防止内存泄漏
}

Tomcat 使用线程池复用线程，请求结束后线程不销毁。RefreshTokenInterceptor.afterCompletion() 中必须调用 UserHolder.removeUser()，否则线程被下一个请求复用时，上一个用户的数据仍残留，导致数据串用（A 的请求读到 B 的用户信息）和内存泄漏（ThreadLocalMap 的 Entry 永远无法被 GC 回收）。

Redis 缓存设计

缓存穿透（Cache Penetration）

定义：查询一个数据库中根本不存在的数据，缓存永远不会命中，每次请求都直接打到数据库。恶意攻击者可以用大量不存在的 ID 发起请求，将数据库打垮。

解决方案对比：

方案 原理 优点 缺点
缓存空值 查 DB 为空时，往缓存写一个空字符串，设短 TTL 简单易实现 占用缓存空间；短时间内仍能穿透
布隆过滤器 预热时将所有合法 ID 写入布隆过滤器，请求前先判断 ID 是否可能存在 内存占用极低，过滤效果好 有误判率（不存在的 ID 可能判断为存在）；新增数据需同步更新过滤器
接口限流/鉴权 对请求参数做格式校验，非法参数直接拒绝 成本最低 只能防格式非法，不能防合法格式的不存在 ID

public <R, ID> R queryWithPassThrough(
        String keyPrefix, ID id, Class<R> type,
        Function<ID, R> dbFallback, Long time, TimeUnit unit) {

    String key = keyPrefix + id;
    String json = stringRedisTemplate.opsForValue().get(key);
    if (StrUtil.isNotBlank(json)) return JSONUtil.toBean(json, type);  // 命中缓存
    if (json != null) return null;  // 命中空值缓存（""），直接返回 null，不打 DB

    R r = dbFallback.apply(id);  // 查 DB
    if (r == null) {
        // 空值写入缓存，TTL=2分钟（不能太长，否则真实数据写入后仍会返回 null）
        stringRedisTemplate.opsForValue().set(key, "", CACHE_NULL_TTL, TimeUnit.MINUTES);
        return null;
    }
    this.set(key, r, time, unit);
    return r;
}

缓存击穿（Cache Breakdown）

定义：某个热点 Key 突然过期，大量并发请求同时发现缓存失效，全部涌入数据库重建缓存，造成 DB 瞬间压力激增。

解决方案对比：

方案 原理 一致性 可用性 适用场景
互斥锁 SETNX 抢锁，只有一个线程重建缓存，其余等待 强一致 低（等待期间阻塞） 对数据一致性要求高
逻辑过期 不设真实 TTL，Value 内存逻辑过期时间，过期后立即返回旧数据并异步重建 弱一致（可能短暂返回旧数据） 高（不阻塞） 热点数据、允许短暂旧数据
乐观锁 缓存值带版本号，重建时 CAS 更新，失败则重试或放弃 较强 较高（无显式阻塞） 并发写缓存冲突概率不高时
永不过期 热点 Key 不设 TTL，通过定时任务或写操作主动更新 视更新策略而定 最高 数据变更频率极低的热点

// tryLock/unLock：用 Redis SETNX 实现，TTL=10秒防止死锁
private boolean tryLock(String key) {
    Boolean flag = stringRedisTemplate.opsForValue()
        .setIfAbsent(key, "1", LOCK_SHOP_TTL, TimeUnit.SECONDS);
    return BooleanUtil.isTrue(flag);
}
private void unLock(String key) { stringRedisTemplate.delete(key); }

// 使用：拿不到锁则休眠 50ms 后递归重试
public Shop queryWithMutex(Long id) {
    String shopJson = stringRedisTemplate.opsForValue().get(CACHE_SHOP_KEY + id);
    if (StrUtil.isNotBlank(shopJson)) return JSONUtil.toBean(shopJson, Shop.class);
    if (shopJson != null) return null;  // 空值缓存

    String lockKey = LOCK_SHOP_KEY + id;
    if (!tryLock(lockKey)) {
        Thread.sleep(50);
        return queryWithMutex(id);  // 自旋重试
    }
    try {
        Shop shop = getById(id);
        if (shop == null) {
            stringRedisTemplate.opsForValue().set(CACHE_SHOP_KEY + id, "", CACHE_NULL_TTL, TimeUnit.MINUTES);
            return null;
        }
        stringRedisTemplate.opsForValue().set(CACHE_SHOP_KEY + id, JSONUtil.toJsonStr(shop), CACHE_SHOP_TTL, TimeUnit.MINUTES);
        return shop;
    } finally {
        unLock(lockKey);  // 无论成功与否，必须释放锁
    }
}

// RedisData.java - 带逻辑过期时间的包装类
public class RedisData {
    private LocalDateTime expireTime;
    private Object data;  // 实际业务数据
}

public <R, ID> R queryWithLogicalExpire(
        String keyPrefix, ID id, Class<R> type,
        Function<ID, R> dbFallback, Long time, TimeUnit unit) {

    String json = stringRedisTemplate.opsForValue().get(keyPrefix + id);
    if (StrUtil.isBlank(json)) return null;  // 未命中（热点数据需提前预热写入）

    RedisData redisData = JSONUtil.toBean(json, RedisData.class);
    R r = JSONUtil.toBean((JSONObject) redisData.getData(), type);

    if (redisData.getExpireTime().isAfter(LocalDateTime.now())) {
        return r;  // 逻辑未过期，直接返回
    }

    // 逻辑已过期：尝试获取互斥锁
    if (tryLock(LOCK_SHOP_KEY + id)) {
        // 拿到锁，开启独立线程异步重建（10线程池）
        CACHE_REBUILD_EXECUTOR.submit(() -> {
            try {
                R r1 = dbFallback.apply(id);
                this.setWithLogicalExpire(keyPrefix + id, r1, time, unit);
            } finally {
                unLock(LOCK_SHOP_KEY + id);
            }
        });
    }
    return r;  // 无论是否拿到锁，都立即返回旧数据，不阻塞
}

互斥锁：强一致，用于对数据准确性要求高的场景（如库存查询）。 逻辑过期：高可用，用于读多写少的热点数据（如商铺详情）。

缓存雪崩（Cache Avalanche）

定义：大量 Key 在同一时间集中过期，或 Redis 服务宕机，导致大量请求同时打到数据库，造成 DB 压力激增乃至崩溃。

解决方案：

  • TTL 加随机偏移：在基础 TTL 上加一个随机值，错开过期时间。本项目不同类型数据使用不同 TTL（验证码 2min、商铺 30min、Token 36000min）天然错开。
  • Redis 集群：部署主从 + 哨兵或 Cluster，避免单点故障导致全量缓存失效。
  • 服务降级与熔断：Redis 不可用时，对非核心接口快速返回兜底数据，避免雪崩压垮 DB。
  • 多级缓存：本地缓存（Caffeine/Guava）+ Redis 双层，Redis 故障时本地缓存兜底。

商铺更新策略（Cache Aside）

更新时先操作 DB，再删除缓存（而非更新缓存），原因：

  1. 缓存写入 vs DB 写入通常不同步，更新缓存容易造成脏数据
  2. 先删缓存会导致 DB 更新失败时缓存为空，下次请求直接穿透到 DB
  3. 先更新 DB + 再删缓存：即使删缓存失败，下次读时 DB 数据已是最新，重建缓存即可

// ShopServiceImpl.update()
@Transactional  // 保证 DB 更新和缓存删除的一致性
public Result update(Shop shop) {
    updateById(shop);  // 1.先更新数据库
    stringRedisTemplate.delete(CACHE_SHOP_KEY + shop.getId());  // 2.再删除缓存

// RedisIDWorker.java
private static final long BEGIN_TIMESTAMP = 1640995200L; // 2022-01-01 00:00:00 UTC
private static final int COUNT_BITS = 32;

public long nextId(String keyPrefix) {
    // 1. 时间戳部分（31位，支持约69年）
    long timestamp = LocalDateTime.now().toEpochSecond(ZoneOffset.UTC) - BEGIN_TIMESTAMP;

    // 2. 序列号部分：按天计数，每日一个 Key，防止单个 Key 无限增长
    String date = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy:MM:dd"));
    long count = stringRedisTemplate.opsForValue().increment("icr:" + keyPrefix + ":" + date);

    // 3. 拼接：时间戳左移32位，与序列号按位或
    return timestamp << COUNT_BITS | count;
}
// 调用示例：Long orderId = redisIDWorker.nextId("order");

优点：

  • 全局唯一、单调递增（同秒内按 count 递增）
  • 不依赖数据库，并发性能高
  • 按天拆分 Key，便于统计每日订单量（icr:order:2025:01:01）

秒杀优惠券系统（核心难点）

秒杀系统需要解决三个核心问题：库存超卖、一人一单、高并发性能。以下是三种方案的演进过程，最终采用 Lua + RabbitMQ 方案。

方案一：纯数据库

直接查库存 → 查是否重复下单 → 扣减库存 → 创建订单，全部在数据库中同步执行。

优点：实现简单，不依赖其他组件。

缺点：

  • 并发时多线程同时读到相同库存，导致超卖
  • 所有操作串行，数据库压力大，QPS 极低
  • 加 synchronized 解决超卖后，性能更差（单线程处理）
1
2
3
4

-- 查库存（并发时多个线程同时通过）
SELECT stock FROM tb_seckill_voucher WHERE voucher_id = ?
-- 扣减（已超卖，无法防止）
UPDATE tb_seckill_voucher SET stock = stock - 1 WHERE voucher_id = ?

方案二：乐观锁 + 分布式锁

扣减库存时加条件 stock > 0 防止超卖；用 Redisson 分布式锁解决一人一单。

优点：

  • CAS 乐观锁解决超卖，并发性好于方案一
  • Redisson 分布式锁（看门狗续期）保证一人一单正确性

缺点：

  • 每次秒杀仍需访问数据库判断库存和重复订单，高并发下 DB 是瓶颈
  • 分布式锁对每个用户串行，大量用户并发时锁竞争激烈
  • 无异步化，接口响应时间与 DB 写入强耦合

方案三：Lua + JVM 阻塞队列

Lua 脚本在 Redis 中原子判断购买资格（库存 & 重复下单），有资格则写入 JVM 内部 ArrayBlockingQueue，异步线程消费创建订单。

优点：

  • 资格判断在 Redis 完成，不访问 DB，接口响应快

  • 队列无持久化、无重试机制，可靠性差
  • 内存队列大小受 JVM 限制，流量突增时有溢出风险

最终实现：Lua + RabbitMQ

用 RabbitMQ 替代 JVM 阻塞队列，解决数据丢失和可靠性问题。

相比方案三的改进：

  • 队列持久化到磁盘，服务重启不丢消息
  • ACK/NACK 确认机制保证消费可靠性
  • 死信队列处理消费失败的订单，防止消息丢失
  • 天然支持集群部署，横向扩展

消息可靠性保障

RabbitMQ 默认不保证消息不丢失，需要从持久化、生产者确认、消费者 ACK 三层配置：

持久化（Broker 重启不丢消息）

三要素必须同时满足：Exchange durable=true、Queue durable=true、消息本身 deliveryMode=PERSISTENT（RabbitTemplate 默认已是持久化投递）。

    publisher-returns: true             # 开启路由失败 Return

@Component
@Slf4j
@RequiredArgsConstructor
public class SeckillOrderConsumer {

    private final IVoucherOrderService voucherOrderService;
    private final RedissonClient redissonClient;

    @RabbitListener(queues = RabbitMQConfig.SECKILL_QUEUE)
    public void handleOrder(VoucherOrder voucherOrder, Channel channel,
                            @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
        Long userId = voucherOrder.getUserId();
        // Redisson 分布式锁：防止网络重试导致的重复消费
        RLock lock = redissonClient.getLock("lock:order:" + userId);
        boolean isLock = lock.tryLock();
        if (!isLock) {
            log.warn("重复消费，跳过：userId={}", userId);
            channel.basicAck(tag, false);
            return;
        }
        try {
            voucherOrderService.createVoucherOrder(voucherOrder);
            channel.basicAck(tag, false);  // 手动 ACK
        } catch (Exception e) {
            log.error("订单创建失败，进入死信队列：{}", voucherOrder.getId(), e);
            channel.basicNack(tag, false, false);  // 不重新入队，转死信
        } finally {
            lock.unlock();
        }
    }

    // 死信队列消费者（处理最终失败订单）
    @RabbitListener(queues = RabbitMQConfig.SECKILL_DL_QUEUE)
    public void handleDeadLetter(VoucherOrder voucherOrder) {
        log.error("秒杀订单进入死信队列，需人工处理：{}", voucherOrder.getId());
        // 可发告警通知（钉钉/企业微信等）
    }
}

@Transactional 生效的前提是通过 Spring 代理对象调用。SeckillOrderConsumer 是独立的 Bean，注入的 voucherOrderService 本身就是 Spring 代理，因此 createVoucherOrder() 的事务能正常生效，无需 AopContext.currentProxy()。若在同一个类中用 this.createVoucherOrder() 调用，会绕过代理导致事务失效——这是 Spring 事务最常见的坑之一。

// BlogServiceImpl.likeBlog()
public Result likeBlog(Long id) {
    Long userId = UserHolder.getUser().getId();
    String key = BLOG_LIKED_KEY + id;  // blog:liked:{blogId}

    // 用 ZSet score 判断是否已点赞（ZSCORE 返回 null 则未点赞）
    Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());

    if (score == null) {
        // 未点赞 → 点赞：DB liked+1，ZSet 存入（score = 当前时间戳）
        boolean isSuccess = update().setSql("liked = liked + 1").eq("id", id).update();
        if (isSuccess) stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
    } else {
        // 已点赞 → 取消：DB liked-1，ZSet 移除
        boolean isSuccess = update().setSql("liked = liked - 1").eq("id", id).update();
        if (isSuccess) stringRedisTemplate.opsForZSet().remove(key, userId.toString());
    }
    return Result.ok();
}

// queryBlogLikes() - 查询 Top5 点赞用户（按时间排序）
public Result queryBlogLikes(Long id) {
    String key = BLOG_LIKED_KEY + id;
    Set<String> top5 = stringRedisTemplate.opsForZSet().range(key, 0, 4);  // score 最小的5个（最早点赞）

    List<Long> ids = top5.stream().map(Long::valueOf).collect(Collectors.toList());
    String idsStr = StrUtil.join(",", ids);
    // 注意：IN 查询结果不保证顺序，需用 ORDER BY FIELD 保持 ZSet 顺序
    List<UserDTO> users = userService.query()
        .in("id", ids)
        .last("order by field(id," + idsStr + ")")
        .list().stream()
        .map(u -> BeanUtil.copyProperties(u, UserDTO.class))
        .collect(Collectors.toList());
    return Result.ok(users);
}

Push 模式 vs Pull 模式对比：

维度 Push（写扩散） Pull（读扩散）
写入时机 发布时推给所有粉丝 用户读取时主动拉取
读性能 高（直接读自己的 Feed ZSet） 低（需遍历所有关注人的最新博文）
写性能 低（大 V 百万粉丝时写放大严重） 高（只写一条）
本项目 ✅ Push 模式 ❌

核心设计总结

场景 技术方案 关键点
登录鉴权 Redis Hash 存 Token BeanUtil 转 Map，所有值 toString（避免类型问题）
Token 刷新 双拦截器链 第一层对所有请求刷新 TTL，第二层才校验登录
缓存穿透 缓存空值 TTL=2min 防止频繁请求不存在 ID 打穿 DB
缓存击穿 逻辑过期 + 异步重建 旧数据先返回，10线程池异步更新
缓存击穿 互斥锁 SETNX 强一致但阻塞，适合对数据准确性要求高的场景
缓存更新 Cache Aside（先 DB 后删缓存） @Transactional 保证原子性
全局 ID Redis INCR + 时间戳位拼接 按天分 Key，单调递增，不可预测
秒杀超卖 Lua 原子脚本 库存检查 + 扣减 + 防重原子完成，资格判断不访问 DB
秒杀一人一单 Redisson 分布式锁 消费者端加锁防止网络重试导致的重复消费，看门狗续期 + 可重入
异步落库 RabbitMQ + 死信队列 持久化存储，ACK/NACK 确保可靠消费，死信队列处理最终失败订单
点赞排行 ZSet（score=时间戳） ZSCORE 判断是否已赞，ZRANGE 取 Top5，ORDER BY FIELD 保持顺序

踩坑与难点

@Transactional 同类调用失效

定位问题：压测秒杀接口时发现，消费端处理订单时数据库出现不完整记录——订单行已插入但库存字段未扣减，且失败后没有回滚。加日志确认两步写库都被调用，怀疑事务未生效。打印事务状态，确认 createVoucherOrder() 执行时根本没有开启事务。

解决尝试：确认 @Transactional 注解存在，排除注解缺失。进一步发现调用方 handleVoucherOrder() 和 createVoucherOrder() 同在 VoucherOrderServiceImpl 中，this.createVoucherOrder() 绕过了 Spring 代理，AOP 切面未介入。

尝试一：自注入 @Autowired private IVoucherOrderService self，通过 self.createVoucherOrder() 调用——可行，但存在循环依赖隐患，代码语义奇怪。

尝试二：在 JVM 队列方案中改用 AopContext.currentProxy() 获取当前代理对象再调用：

1
2

proxy = (IVoucherOrderService) AopContext.currentProxy();
proxy.createVoucherOrder(voucherOrder);  // 通过代理调用，事务生效

最终效果：演进到 RabbitMQ 方案时，将消费逻辑抽为独立的 SeckillOrderConsumer Bean，注入的 voucherOrderService 本身就是 Spring 代理，同类调用问题从架构上彻底消除，代码更清晰，事务一直正常生效，不完整订单记录问题不再出现。

ThreadLocal 数据残留导致串用

定位问题：测试时发现某接口偶发返回了其他用户的信息（如用户名、头像），概率约 1/100，刷新后恢复正常。最初怀疑是 Redis 缓存 key 碰撞，排查 RedisConstants key 设计后排除。在拦截器中添加线程 ID 日志，发现两次不同用户的请求复用了同一线程，且前一次请求结束后 ThreadLocal 中的 UserDTO 没有被清空。

解决尝试：查看 RefreshTokenInterceptor 发现 afterCompletion() 方法只做了 TTL 刷新，漏掉了 UserHolder.removeUser()。尝试在每个 Controller 方法末尾手动 remove——方法出口太多（包括异常路径），无法全面覆盖，方案不可行。

最终效果：在 RefreshTokenInterceptor.afterCompletion() 中加入 UserHolder.removeUser()。afterCompletion 是 Spring MVC 拦截器的最终钩子，无论请求正常结束还是抛出异常都会执行，保证每次请求结束后当前线程的用户信息必然被清空。上线后串用现象完全消失，线程 ID 日志也确认每次请求处理前 ThreadLocal 中均为空值。

Feed 流 offset 分页数据漂移

定位问题：用户反馈下拉 Feed 流时偶尔看到重复博文，或某篇博文找不到了。在测试环境稳定复现：分页查询期间发布新博文，第 2 页第一条内容会与第 1 页最后一条重复，同时另一篇博文被跳过永远不会出现。

解决尝试：方案一：客户端缓存第一页数据，翻页时只查增量——实现复杂，且被跳过的问题仍未解决。方案二：查询时带快照时间戳参数，固定查询时刻——需要额外的存储或索引，代价大，且 ZSet 不直接支持时间点快照查询。

最终效果：改用 ZSet 的 ZRANGEBYSCORE + score 游标分页。每次请求记录返回结果中的最小时间戳（minTime）和同分项偏移量（offset），下次查询以 max=minTime, offset=offset 为起点，无论期间新数据如何插入，游标后的相对顺序不变。改造后 Feed 翻页完全正确，经多轮并发发帖测试再无重复或跳过，ScrollResult 的游标字段被前端复用至今稳定。

秒杀超卖（并发读同一库存）

定位问题：第一版秒杀接口以 1000 并发压测，库存设为 100 张，结束后数据库中生成了 120+ 个订单，超卖 20%。查询订单详情，多个不同用户的订单在同一毫秒内创建，说明是并发读写竞争而非计数 bug。

解决尝试：

方案一：synchronized(this) 锁住 seckillVoucher() 方法——单机有效，但集群环境下各节点 JVM 独立，锁不跨进程，分布式部署后超卖依然存在。

方案二：SQL 乐观锁 UPDATE ... WHERE stock > 0——DB 层保证不超卖，但 1000 并发全部打到数据库，大量 update 失败需要重试，数据库压力极大，接口响应时间劣化至 2s+，不可用。

最终效果：将库存检查和扣减移入 Lua 脚本，在 Redis 中原子执行——判断 stock <= 0 返回失败码，否则 DECR 扣减，并将用户 ID 写入 Set 实现一人一单。Redis 单线程执行 Lua 脚本，整个操作不可分割，彻底消除并发竞争。DB 只接收 RabbitMQ 异步投递的最终落库消息，压测后订单数量与库存完全吻合，接口 P99 响应时间从 2s 降至 50ms 以内。
