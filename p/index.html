虚拟内存和物理内存

页 页框 页表

::页是页表的管理对象:: 这里的页特指虚拟页，它是进程虚拟地址空间的最小划分单位。页表不是为某一个虚拟页单独存在，而是为整个进程的虚拟地址空间服务，其核心功能就是记录进程中需要映射的虚拟页，分别对应哪一个物理页框。只有存在映射需求的虚拟页，才会在页表中拥有对应的页表项；未使用或被换出的虚拟页，在页表中无有效页表项或标记为无效状态。 ::页表是虚拟页与物理页框的映射桥梁:: 虚拟页本身不对应实际物理存储，必须通过页表的页表项，才能建立与物理页框的关联。页表项中存储的物理页框号，是虚拟页能够被 CPU 访问的关键；同时页表项中的权限位、有效位等字段，还会直接管控虚拟页的访问规则（如可读可写、只读）和映射状态（如已映射、换出到磁盘）。

内存空间分配

已知有100MB空间，A操作使用了30MB，然后B操作使用了35MB，然后删除了A操作的占用，然后C操作使用了45MB，然后删除了B操作的占用，求最大空闲空间

关键步骤：C操作时，连续内存不够用，被分成了B操作前面的30MB和B操作后的15，因此最后总空间位55MB，最大空闲连续空间位35MB

Windows相关

本地安全与域安全策略

::核心定位与作用范围关系:: Windows 本地安全策略与域安全策略是层级互补的安全管控体系，本地安全策略作用于单台计算机，管控本地账户与单机安全规则，适配工作组等分散环境；域安全策略以域控制器为核心，可集中管控域内所有成员设备的安全配置，适配中大型企业的统一管理需求。二者仅在计算机加入域时产生交互，未加入域的设备仅受本地安全策略约束。 ::优先级与执行规则关系:: 二者的核心执行原则是域安全策略优先于本地安全策略：计算机加入域后，若两套策略存在冲突，域策略会覆盖本地策略并锁定相关配置项；若策略无冲突，则会合并生效。同时，域策略不会完全替代本地策略，未被域策略定义的本地安全规则仍可生效，可作为域策略的补充满足单台设备的个性化安全需求；而域控制器的本地策略会升级为域控制器安全策略，优先级高于普通域安全策略。 ::域策略的优先级更高，是为了实现 “全局统一管控”，而不是因为范围大小；同层级的策略里，才遵循 “小范围优先于大范围” 的逻辑。::

类变量和静态变量

实际为同一概念 ::静态变量::：通过static关键字修饰的变量（语法层面的定义）； ::类变量::：属于 “类本身” 而非类的实例对象的变量（语义层面的定义）。

类变量作用域

::误区 1::：“类变量作用域和实例绑定”❌ 错误：类变量不归属于实例，哪怕没有创建任何对象，只要类加载完成，就可通过类名.变量名访问（比如Math.PI，无需 new Math ()）；✅ 正确：作用域是 “类本身”，实例仅能间接访问，而非决定作用域。 ::误区 2::：“静态方法能访问所有类的变量”❌ 错误：静态方法可访问「当前类的类变量」，但访问「其他类的类变量」仍受访问修饰符约束（比如其他类的 private 静态变量无法访问）；✅ 正确：静态方法的访问权限 ≠ 类变量的作用域，作用域最终由变量自身的修饰符决定。 ::误区 3::：“类变量作用域随方法执行结束消失”❌ 错误：类变量存储在方法区，生命周期和类绑定，而非方法；哪怕所有方法执行完毕，类变量仍存在（除非类被卸载）。

异常处理参数作用域

一、异常抛出的作用域：由方法 / 代码块的可见性 + 声明约束 ::检查型异常::（如 IOException）：抛出范围受throws声明约束，仅能在声明了该异常的方法 / 子类重写方法中抛出（子类重写时不能抛出父类未声明的检查型异常，符合里氏代换原则）； ::运行时异常::（如 NullPointerException）：无显式声明约束，可在任意代码块（方法、循环、分支）中抛出，作用域覆盖当前执行的代码路径； ::异常的传播作用域::：未捕获的异常会沿调用栈向上传播（从当前方法→调用方方法→上层调用方），直到被捕获或终止线程，传播范围不跨线程（线程内独立传播）。 二、异常捕获的作用域：仅覆盖「try 代码块内的执行逻辑」 基础规则：catch块仅能捕获对应try块内执行过程中抛出的、匹配异常类型（或子类类型）的异常，try块外的异常无法被该catch捕获； 示例： java 运行 public void test() { int a = 1; try { int b = a / 0; // try块内异常可被捕获 } catch (ArithmeticException e) { e.printStackTrace(); } int c = a / 0; // try块外异常无法被上述catch捕获，直接抛出 } 多 catch 块的作用域优先级：子类异常的 catch 块需放在父类异常之前（如先 catch NullPointerException，再 catch Exception），否则子类异常会被父类 catch 块覆盖，失去针对性处理的意义； finally 的作用域：finally 块无论 try 块是否抛出异常、catch 是否捕获，都会执行（除非 JVM 终止），作用域是 try-catch 流程的收尾，常用于释放资源（如关闭流、连接），其执行不受异常处理结果影响。

字符串String由哪两个包定义

::String 类的核心定义包:: Java 中String类的唯一定义包是java.lang.String，该包是 Java 核心基础包，无需手动导入即可直接使用String类，其所有核心属性（如存储字符的value数组）、核心方法（如equals()、substring()）均在此包中实现，是String类存在的基础。 ::String 类的核心关联包:: 不存在第二个定义String类的包，但java.nio.charset是String功能的核心关联包 ——String的编码、解码能力（如getBytes(Charset)、通过字符集构造String）完全依赖该包的Charset类，是处理String字符编码场景的关键支撑，也因此常被混淆为 “第二个定义包”。 三、异常处理后的影响作用域：局部生效，不跨上下文 ::方法内处理::：catch 块内的修复逻辑（如赋值、重试）仅作用于当前方法的局部变量 / 执行流程，不会影响其他方法的变量或线程； ::全局处理::（如 Thread.setUncaughtExceptionHandler）：仅作用于当前线程，不同线程的未捕获异常需单独配置处理器，无法跨线程生效； ::类级别的异常处理::（如 Spring 的 @ControllerAdvice）：作用域覆盖指定包 / 类下的所有控制器方法，超出该范围的非控制器方法异常无法被处理。

Java标识符命名规范

Java 标识符命名需同时遵守语法强制规则和编码约定规范：语法上，标识符只能由字母（A-Z/a-z）、数字（0-9）、下划线（_）、美元符（$）组成，不能以数字开头，不能使用 Java 关键字（如 int class）或保留字（如 goto const），且区分大小写；编码约定上，包名采用全小写多级结构（如 com.example.demo），类名和接口名用大驼峰命名法（每个单词首字母大写，如 UserService），方法名和变量名用小驼峰命名法（首单词小写、后续单词首字母大写，如 getUserName orderId），常量名需全大写且单词间用下划线分隔（如 MAX_AGE），枚举类名用大驼峰、枚举值全大写，注解名推荐大驼峰命名。此外，语法上允许使用中文但编码规范严格禁止，美元符仅限自动生成代码使用，手动编码应避免。

设计原则

里氏代换原则(LSP)

::子类必须能够替换掉其父类，且不影响程序的正确性:: 遵循里氏代换原则的核心好处是，以子类可安全替换父类且不破坏程序原有逻辑为基础，既保障了程序的稳定性与行为可预测性，避免因子类篡改父类契约引发隐蔽 bug；又能提升父类通用逻辑的复用率，减少冗余开发；同时还能支撑开闭原则，让新增子类时无需修改原有代码，降低系统扩展与维护成本，也能简化团队协作中对继承体系的理解与使用门槛。

IOC自动装配方式

::一、按名称装配（byName）:: 按属性名匹配容器中同名 Bean，XML 配置autowire=“byName”，注解常用@Resource(name=“bean名”)或@Autowired+@Qualifier，匹配精准，避免同类型 Bean 冲突。 ::二、按类型装配（byType）:: 按属性类型匹配同类型 Bean，XML 配置autowire=“byType”，注解直接用@Autowired，简化配置，但要求同类型 Bean 唯一，否则需用@Qualifier指定名称。 ::三、构造器装配（Constructor）:: 通过构造器注入依赖，XML 配置autowire=“constructor”，注解可省略@Autowired，优先按参数类型匹配，保证依赖不可变，是 Spring 推荐方式。 ::四、方法装配（Method）:: 通过方法调用注入，含 Setter 方法（@Autowired标注 setXxx ()）、工厂方法、@Bean方法（Spring Boot 主流）三种形式，灵活适配自定义 Bean 装配场景。 ::构造器方法用@lazy，其他三个方法均可三级缓存解决循环依赖::

IO相关

流的传递方式

一、核心维度：流传递的 5 大分类（按特征划分） ::按传输方向（基础）::： 输入流：数据从外部载体（文件、网络、硬件）流向程序 / 内存（如FileInputStream、BufferedReader）； 输出流：数据从程序 / 内存流向外部载体（如FileOutputStream、BufferedWriter）； 单个流对象均为单向传输，双向传输需同时创建输入 + 输出流。 ::按传输粒度（最小单元）::： 字节流：以 8 位byte为单元，适配所有数据（二进制文件、图片、视频、文本），如InputStream/OutputStream体系； 字符流：以 16 位char为单元，专为文本优化（自动处理编码），是字节流 + 编码转换的封装，如Reader/Writer体系（字符传递是其核心场景）。 ::按传输模式（是否阻塞::）： 同步流：调用读写方法后线程阻塞，直到传输完成（默认方式，如FileReader.read()），逻辑简单但高并发易卡顿； 异步流：调用后线程不阻塞，通过回调 / 事件接收结果（如 Java NIO 的AsynchronousFileChannel），适配大文件 / 高并发场景。 ::按传输载体（应用场景）：:: 文件流：磁盘与内存间传输（如FileInputStream）； 网络流：基于 Socket 的网络节点间传输（TCP 为有序字节流，UDP 为无连接数据报流）； 内存流：内存缓冲区内部传输（如ByteArrayInputStream），无 IO 开销； 管道流：进程 / 线程间通信（如PipedReader）； 压缩流：边传输边压缩 / 解压（如GZIPInputStream）。 ::按优化方式（是否缓冲）::： 非缓冲流：直接读写底层载体，每次操作触发 IO，效率低（如FileReader）； 缓冲流：先批量读取到内存缓冲区，减少 IO 次数（如BufferedReader），是最常用的优化方式。 二、通用规则：所有流传递的共性特征 ::顺序性::：多数流为 “顺序流”，按写入 / 读取顺序传输，随机访问需特殊实现（如RandomAccessFile）； ::资源释放::：流关联底层资源（文件句柄、网络连接），需手动 / 自动（try-with-resources）关闭，避免泄漏； ::关联性::：字符流是字节流的封装，缓冲流是普通流的包装，可多层嵌套优化传输（如BufferedReader包装FileReader）。

符号链接和硬链接

::硬链接::：多个文件名（指针）直接指向同一个文件本体（inode + 数据块）系统里的文件本体是通过 inode 标识的，文件名只是给用户看的 “访问入口”。创建硬链接，就是新增一个 “入口”（文件名），和原文件名共用同一个文件本体。所以这些文件名之间没有主次之分，删除其中一个，只是少了一个入口，文件本体只要还有其他入口（链接计数 > 0）就不会消失。 ::符号链接::：一个文件名（指针）指向另一个文件名（指针）符号链接本身是一个独立的小文件，里面只存了目标文件的路径字符串。当你访问符号链接时，系统会先根据这个路径找到目标文件名，再通过目标文件名找到文件本体。这就是为什么删除目标文件后，符号链接会变成 “悬空链接”—— 它指向的那个 “指针”（目标文件名）没了，自然找不到文件本体。

public class MinBitwiseAndForLargeNumbers {

/**
 * 从数组中选取k个数，使得它们的按位与结果最小（支持2^60范围内的数字）
 * @param nums 输入的数字数组（使用long类型）
 * @param k 需要选取的数字个数
 * @return 最小的按位与结果
 */
public static long findMinBitwiseAnd(long[] nums, int k) {
    // 处理边界情况
    if (k <= 0 || k > nums.length) {
        return 0;
    }
    
    long resultMask = 0L;
    
    // 从最高位（第60位）到最低位（第0位）尝试构建掩码
    for (int bit = 60; bit >= 0; bit--) {
        // 尝试将当前位设置为1
        long candidateMask = resultMask | (1L << bit);
        int count = 0;
        
        // 统计有多少个数满足 (num & candidateMask) == candidateMask
        for (long num : nums) {
            if ((num & candidateMask) == candidateMask) {
                count++;
            }
        }
        
        // 如果有足够多的数满足条件，更新掩码
        if (count >= k) {
            resultMask = candidateMask;
        }
    }
    
    // 计算最小的按位与结果：所有不在resultMask中的位都置1
    // 注意：这里只处理0~60位，避免高位干扰
    long minResult = 0L;
    for (int bit = 60; bit >= 0; bit--) {
        if ((resultMask & (1L << bit)) == 0) {
            minResult |= (1L << bit);
        }
    }
    
    return minResult;
}

public static void main(String[] args) {
    // 测试示例1：包含大数的情况
    long[] nums1 = {1L << 50, 1L << 55, (1L << 50) | (1L << 55), 123456789L};
    int k1 = 2;
    System.out.println("数组中选取 2 个数的最小按位与结果: " 
                       + findMinBitwiseAnd(nums1, k1));
    
    // 测试示例2：常规数字
    long[] nums2 = {3L, 6L, 7L, 9L, 12L};
    int k2 = 2;
    System.out.println("数组 [3, 6, 7, 9, 12] 中选取 2 个数的最小按位与结果: " 
                       + findMinBitwiseAnd(nums2, k2));
}

}

// 如果不是 String 类型，或者不匹配任何条件，返回 null

Bug的生命周期

Bug 完整生命周期（各阶段详细说明） ::新建（New）::新建是 Bug 生命周期的起点，指测试人员、开发人员或用户发现软件异常后，在缺陷管理工具（如 Jira、Bugzilla）中提交 Bug 单的阶段。提交时需完整记录 Bug 的核心信息，包括复现步骤、测试环境、预期结果与实际结果、截图 / 日志等，确保 Bug 描述清晰可追溯；此时 Bug 状态为 “新建”，等待专人审核，若信息不全可能被打回补充，这是后续所有处理环节的基础。 ::确认（Verified/Confirmed）::确认阶段由 Bug 负责人（如测试组长、开发负责人）对新建的 Bug 进行审核验证，核心是判断 Bug 是否为真实存在的问题：若验证后确认是代码缺陷、逻辑错误等有效问题，将状态转为 “确认”；若判定为需求理解偏差、操作失误、环境问题等无效 Bug，会直接标记为 “关闭 - 无效”，并备注原因反馈给提交人；这一阶段能过滤无效 Bug，避免开发资源浪费。 ::分配（Assigned）::分配阶段是将确认后的有效 Bug 定向指派给对应的开发人员，明确修复责任人。负责人需根据 Bug 所属模块、业务范围或代码归属，精准分配给相关开发工程师，同时可标注 Bug 优先级（高 / 中 / 低）和严重程度（致命 / 严重 / 一般 / 轻微），让开发人员清晰知晓修复优先级；若分配错误，可重新调整责任人，确保 Bug 能被对应领域的开发人员处理。 ::修复中（In Progress/Fixing）::修复中阶段是开发人员接手已分配的 Bug 后，开展问题定位、分析根因并编写修复代码的过程。开发人员需先复现 Bug，通过日志、调试工具等排查问题根源（如逻辑漏洞、参数错误、兼容性问题等），再制定修复方案并修改代码；期间可更新 Bug 备注，记录修复思路、遇到的难点等，若需延期修复，需标注原因和预计完成时间，确保进度可追踪。 ::已修复（Fixed/Resolved::）已修复阶段是开发人员完成代码修改、自测验证 Bug 已解决后，将 Bug 状态更新为 “已修复” 的环节。此时需附带修复相关信息，如代码提交记录、自测结果、修复范围等，同时将 Bug 回传给测试人员；若修复过程中发现 Bug 无法复现或无需修复（如需求变更），也可在该阶段标注 “已解决 - 无需修复”，并说明具体原因。 ::回归测试（Retest/Reopened）::回归测试阶段由测试人员针对修复后的版本，按照原复现步骤验证 Bug 是否真正解决。若测试后 Bug 未复现，说明修复有效，将状态转为 “验证通过”；若 Bug 仍存在、复现步骤不变，或修复后引入新问题，会将 Bug 状态打回 “重新打开（Reopened）”，并备注未修复的原因或新问题的表现，Bug 会再次回到 “分配” 阶段，进入新一轮修复流程。 ::关闭（Closed）::关闭是 Bug 生命周期的最终闭环阶段，仅在回归测试通过后执行：测试人员确认 Bug 完全修复，且无衍生问题，将 Bug 标记为 “关闭”，并归档留存；若 Bug 因特殊原因无法修复（如兼容成本过高、业务调整无需修复），也可标注 “关闭 - 无法修复”“关闭 - 需求变更” 等具体原因后归档；关闭后的 Bug 若后续复现，可重新激活，再次进入生命周期流程。

算法

给n个数，取部分数的和为k的倍数，要求数最大

public class Main { /** * 找到和为k的倍数的最大和值 * @param nums 输入数组 * @param k 目标倍数 * @return 满足条件的最大和值（无满足条件的子集返回0） / /* * 找到数组中选取若干数，使其和为k的倍数的最大和值 * @param nums 输入的整数数组 * @param k 目标倍数 * @return 满足条件的最大和值（若无满足条件的非空子集则返回0） */ public static int findMaxSumMultiple(int[] nums, int k) { // dp[i]：表示选取某些数后，它们的和对k取模等于i时的最大和值 // 数组大小为k，对应余数0到k-1的所有可能情况 int[] dp = new int[k];

    // 初始化dp数组：将所有余数状态初始化为Integer.MIN_VALUE，表示初始时不可达
    // 这是为了区分"和为0"（余数0且和为0）和"不可达"（余数i无法实现）
    for (int i = 0; i < k; i++) {
        dp[i] = Integer.MIN_VALUE;
    }

    // 初始状态：选取0个数时，和为0，余数为0，所以dp[0] = 0
    // 这是动态规划的起点
    dp[0] = 0;

    // 遍历数组中的每一个数字，逐个考虑是否选取该数字
    for (int num : nums) {
        // 关键：创建dp数组的副本temp，用于存储更新后的状态
        // 这样可以避免在同一轮迭代中重复使用当前数字多次
       ::int[] temp = dp.clone();::

        // 遍历所有可能的余数状态（0到k-1）
        for (int i = 0; i < k; i++) {
            // 如果当前余数状态i是可达的（不是初始的MIN_VALUE）
            if (dp[i] != Integer.MIN_VALUE) {
                // 计算当前数字num对k取模的余数
                // (num % k + k) % k 是为了处理num为负数的情况，确保余数非负
                int remainder = (num % k + k) % k;

                // 计算加入当前数字后新的余数：(当前余数i + 数字余数remainder) % k
                int newMod = (i + remainder) % k;

                // 计算加入当前数字后的新和值：当前和值dp[i] + 当前数字num
                int newSum = dp[i] + num;

                // 如果新和值比temp中记录的newMod状态的和值更大，则更新
                // 这保证了每个余数状态存储的都是最大的可能和值
                if (newSum > temp[newMod]) {
                    temp[newMod] = newSum;
                }
            }
        }

        // 将更新后的temp数组赋值给dp，完成一轮状态转移
        dp = temp;
    }

    // 最终结果：dp[0]存储的是和为k的倍数（余数0）的最大和值
    // 如果dp[0] > 0，返回该值；否则返回0（表示只有空集满足条件）
    return dp[0] > 0 ? dp[0] : 0;
}

public static void main(String[] args) {
    // 测试用例1
    int[] nums1 = {3, 1, 2, 4, 5};
    int k1 = 3;
    System.out.println("测试用例1:");
    System.out.println("数组: [3, 1, 2, 4, 5], k = 3");
    System.out.println("最大和值: " + findMaxSumMultiple(nums1, k1)); // 输出: 15 (3+1+2+4+5=15)
    System.out.println();

    // 测试用例2

    int k2 = 4;
    System.out.println("测试用例2:");
