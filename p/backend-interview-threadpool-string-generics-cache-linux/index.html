Java 相关

线程池

线程池 7 个核心参数

  1. corePoolSize：核心线程数。
  2. maximumPoolSize：最大线程数。
  3. keepAliveTime：非核心线程的闲置超时时间。
  4. unit：keepAliveTime 的时间单位。
  5. workQueue：任务阻塞队列。
  6. threadFactory：用于创建线程的工厂。
  7. handler：拒绝策略，当任务队列已满且线程数达到最大时执行。

线程池任务堆积、队列过大的解决方式

  • 扩容线程池：适当调大 maximumPoolSize（需结合 CPU 核心数，避免线程过多导致上下文切换开销）。
  • 优化队列：使用有界队列（如 ArrayBlockingQueue）并设置合理容量，配合合适的拒绝策略（如 CallerRunsPolicy，让提交者线程自己执行任务，从而减慢任务提交速度）。
  • 任务降级：在任务量过大时，非核心任务可以降级处理，例如记录日志后续补偿，而不是直接进入线程池。
  • 优化任务处理效率：简化任务逻辑，将 IO 密集型操作异步化，减少单个任务的执行时间。
  • 监控告警：实时监控队列长度、任务执行耗时等指标，超过阈值时触发告警或动态扩容。

扩大内存能否解决线程池问题

扩大内存不能从根本上解决线程池任务执行慢和堆积问题。

  • 任务执行慢：核心原因通常是线程资源不足、任务本身耗时（如 IO 阻塞、CPU 密集型计算）或依赖的外部服务响应慢，这与内存大小无直接关系。
  • 任务堆积：本质是任务提交速度超过处理速度。扩大内存仅能让阻塞队列容纳更多任务，治标不治本，最终仍会耗尽内存或触发拒绝策略。

正确思路：应从优化任务效率（异步化、减少锁竞争）、合理配置线程池参数（如根据任务类型调整核心/最大线程数）、引入限流削峰等角度解决。

如何让线程池任务运行更快

核心是提升单任务执行效率和优化线程池资源利用率。

  • 优化任务本身
    • 减少任务内阻塞，如使用 NIO 或异步 IO 替代同步阻塞 IO。
    • 避免或优化锁竞争，如缩小锁粒度、使用 ConcurrentHashMap 或 CAS 操作替代 synchronized。
    • 将长任务拆分为多个并行的短任务。
  • 提升线程利用率
    • 按任务类型隔离线程池，例如将 CPU 密集型和 IO 密集型任务分配到不同的线程池，避免互相影响。
    • 合理设置 keepAliveTime，避免线程频繁创建和销毁。
    • 使用 ThreadLocal 缓存线程私有资源（如 SimpleDateFormat），减少重复创建开销。
  • 利用硬件资源
    • 对于计算密集型任务，可适当增加线程数以利用多核 CPU。
    • 使用本地缓存（如 Caffeine、Guava Cache）减少重复计算或远程调用。

String 相关

String 为什么不可变

String 被设计为不可变（Immutable）主要是为了效率和安全。

  • 字符串常量池（String Pool）：Java 中大量 String 对象可以被复用。如果 String 可变，修改一个 String 会影响所有引用它的地方，常量池将失去意义。
  • 线程安全：不可变对象天然是线程安全的，无需额外同步措施即可在多线程环境中共享。
  • 适用于 HashMap 的 Key：String 常被用作 HashMap 的键。如果 String 可变，其 hashCode() 值可能改变，导致无法在 HashMap 中正确定位到原来的值。
  • 安全性：在网络连接、文件路径等场景，如果 String 可变，可能在安全校验通过后被恶意修改，引发安全漏洞。

String 能通过反射获取并修改底层数组吗

可以。尽管 String 的底层数组（JDK 9 前为 char[]，后为 byte[]）是 private final 的，但通过反射可以绕过这些限制。

  1. 获取 value 字段：使用 String.class.getDeclaredField("value") 获取私有字段。
  2. 破解访问限制：调用 field.setAccessible(true) 来允许访问私有成员。
  3. 修改数组内容：获取底层数组引用后，直接修改其元素。这会导致所有共享该 String 对象的地方（包括字符串常量池）都受到影响。

这种行为严重破坏了 String 的不可变性设计，可能导致整个应用出现无法预料的错误，严禁在生产代码中使用。

泛型

泛型擦除

泛型擦除是 Java 泛型的一种实现机制。它指的是在编译阶段，泛型的类型参数信息会被擦除，替换为其原始类型（Raw Type），以兼容 JDK 5 之前的代码。

  • 编译时：编译器会检查泛型的类型约束。例如，List<String> 中只能添加 String。同时，编译器会将类型参数 T 替换为它的上界（如 T extends Number 替换为 Number），如果没有指定上界则替换为 Object。
  • 运行时：JVM 看到的是不带类型参数的原始类型。例如，List<String> 和 List<Integer> 在运行时都被看作是 List 类型。

带来的影响：

  • 编译器会在读取泛型对象的地方自动插入强制类型转换代码，以保证类型安全。
  • 为了解决类型擦除后多态方法签名不匹配的问题，编译器会生成桥接方法（Bridge Method）。
  • 基本类型（如 int）不能作为泛型参数，必须使用其包装类（如 Integer）。

泛型在编译和运行阶段的处理

简单来说，泛型是 “编译时很具体，运行时很模糊”。

  • 编译阶段：编译器严格执行类型检查。当你写 List<String> list = new ArrayList<>(); 并尝试 list.add(123); 时，编译器会报错。
  • 编译后：泛型信息被擦除，List<String> 变成 List。同时，编译器会在 list.get(0) 这类读取操作处自动插入 (String) 强制转换。
  • 运行阶段：JVM 只知道这是一个 List，但由于编译器已经保证了存入的都是 String，并且在读取时进行了类型转换，因此在功能上是类型安全的。

缓存相关

使用 HashMap 实现本地缓存

直接使用 HashMap 作为本地缓存是最简单的方式，但功能非常有限，存在明显缺点：

  • 无过期策略：缓存项会永久存储在内存中，除非手动移除，否则可能导致内存泄漏（OOM）。
  • 无淘汰策略：当缓存达到容量上限时，无法自动淘汰旧的或不常用的缓存项。
  • 线程不安全：在多线程环境下直接使用 HashMap 会有并发问题，需要使用 ConcurrentHashMap 代替。

在生产环境中，推荐使用成熟的本地缓存框架，如 Caffeine 或 Guava Cache。它们提供了丰富的过期策略（基于时间、大小、引用）、淘汰算法（如 LRU、LFU）和统计功能。

本地缓存与 Redis 缓存优缺点对比

本地缓存（Local Cache）
  • 优点
    • 极致性能：数据在应用进程内存中，无网络 IO，延迟极低（微秒级）。
    • 实现简单：无需部署额外服务，易于集成。
  • 缺点
    • 存储受限：受单个应用实例的内存限制，无法缓存大量数据。
    • 数据不一致：在分布式环境中，各实例的缓存独立，难以保证数据一致性。
    • 缺乏分布式特性：不支持跨服务共享、分布式锁等功能。
Redis 缓存（分布式缓存）
  • 优点
    • 分布式共享：作为独立服务，可被多应用实例共享，保证数据一致性。
    • 大容量存储：可独立扩容，支持海量数据存储。
    • 功能丰富：支持持久化、多种数据结构、发布订阅、分布式锁等高级功能。
    • 高可用：可通过主从、哨兵、集群模式实现高可用。
  • 缺点
    • 性能开销：依赖网络 IO，访问延迟比本地缓存高（毫秒级）。
    • 部署维护成本：需要独立部署和监控，增加了系统复杂度。
